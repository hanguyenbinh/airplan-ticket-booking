1️⃣ Vấn đề về Rebalancing & Consumer

🔄 Stop-the-World Rebalance
Vấn đề: Toàn bộ consumer dừng xử lý để chia lại partition
⏰ Khi nào xảy ra:
  • Thêm hoặc bớt consumer trong một group
  • Consumer crash và cố khôi phục
  • Consumer timeout (session.timeout.ms) quá ngắn
  • Sử dụng RoundRobinAssignor hoặc RangeAssignor (mặc định)
✅ Cách giải quyết:
  • Chuyển sang CooperativeStickyAssignor - chỉ di chuyển các partition cần thiết
  • Tăng session.timeout.ms (mặc định 10s, tăng lên 30-45s)
  • Tăng heartbeat.interval.ms để consumer gửi heartbeat thường xuyên hơn
  • Tối ưu hóa logic xử lý consumer để nó không vượt quá max.poll.interval.ms
# Java/Spring Boot spring.kafka.consumer.properties.partition.assignment.strategy=org.apache.kafka.clients.consumer.CooperativeStickyAssignor # Python - confluent-kafka 'partition.assignment.strategy': 'cooperative-sticky'
🔗 Rebalance Cascade
Vấn đề: Rebalance kéo dài làm timeout, gây ra đợt rebalance tiếp theo liên tục
⏰ Khi nào xảy ra:
  • Nút Kafka bị chậm hoặc quá tải (CPU/IO)
  • Mạng chập chờn giữa client và broker
  • Consumer xử lý message quá lâu và không kịp gửi heartbeat
  • Cấu hình timeout quá ngắn với logic xử lý nặng
✅ Cách giải quyết:
  • Kiểm tra metrics: CPU, Memory, Disk I/O của broker
  • Nâng cấp hardware nếu cần (CPU, RAM)
  • Tách các logic nặng ra thread riêng để consumer kịp gửi heartbeat
  • Sử dụng KEDA (Kubernetes) để auto-scale dựa trên Consumer Lag thay vì chỉ CPU/RAM
  • Tăng max.poll.interval.ms (mặc định 5 phút, tăng lên 10-15 phút nếu logic xử lý nặng)
# Cấu hình Kafka Consumer tối ưu cho Rebalance session.timeout.ms=45000 heartbeat.interval.ms=3000 max.poll.interval.ms=600000 # 10 phút max.poll.records=100 # Giảm số record để xử lý nhanh hơn
📈 Consumer Lag
Vấn đề: Tốc độ xử lý của consumer chậm hơn tốc độ đẩy dữ liệu của producer
⏰ Khi nào xảy ra:
  • Traffic từ producer vượt ngưỡng xử lý của consumer
  • Logic xử lý consumer quá phức tạp (call API bên ngoài, DB query nặng)
  • Số lượng consumer ít hơn số partition
  • Consumer xảy ra rebalance liên tục
  • Database hoặc service phía sau consumer bị quá tải
✅ Cách giải quyết:
  • Tăng số lượng consumer instances (= số partition)
  • Sử dụng KEDA để auto-scale khi consumer lag > ngưỡng
  • Tối ưu hóa logic xử lý: batch processing, async calls, caching
  • Tăng fetch.min.bytes và fetch.max.wait.ms để nhận được batch lớn hơn
  • Thêm retry logic với exponential backoff cho các external calls
  • Monitor consumer lag bằng Kafka Lag Exporter hoặc Prometheus
# KEDA ScaledObject cho Kafka apiVersion: keda.sh/v1alpha1 kind: ScaledObject metadata: name: kafka-consumer-scaler spec: scaleTargetRef: name: kafka-consumer-app minReplicaCount: 1 maxReplicaCount: 20 # = số partition triggers: - type: kafka metadata: bootstrapServers: kafka:9092 consumerGroup: my-group topic: my-topic lagThreshold: "10000" # Scale up khi lag > 10k messages
☠️ Poison Pill Message
Vấn đề: Một tin nhắn bị lỗi khiến consumer crash mỗi khi thử đọc lại (vòng lặp vô tận)
⏰ Khi nào xảy ra:
  • Message format không hợp lệ (JSON bị hỏng, encoding sai)
  • Consumer code không xử lý exception, throw error liên tục
  • Message có schema không tương thích với schema của consumer
  • External service mà consumer phụ thuộc bị down, consumer retry vô tận
✅ Cách giải quyết:
  • Thêm Dead Letter Queue (DLQ) để gửi message lỗi vào topic riêng
  • Xử lý exception một cách graceful, skip message lỗi và log
  • Sử dụng schema registry để validate message trước xử lý
  • Thêm metrics/alerting để phát hiện khi consumer crash liên tục
  • Implement retry logic với circuit breaker cho external calls
try { ConsumerRecord<String, String> record = ... processMessage(record); } catch (Exception e) { // Gửi vào DLQ thay vì crash producer.send(new ProducerRecord<>("my-topic-dlq", record.value())); logger.error("Failed to process message, sent to DLQ", e); // KHÔNG throw exception - continue processing }
🐢 Slow Processing
Vấn đề: Logic xử lý trong consumer quá nặng làm trigger max.poll.interval.ms
⏰ Khi nào xảy ra:
  • Consumer xử lý message lâu hơn max.poll.interval.ms (mặc định 5 phút)
  • Có blocking I/O (DB query, HTTP call) trong message processing
  • Không có proper resource limit (CPU throttle)
  • Consumer xử lý một batch quá lớn cùng lúc
✅ Cách giải quyết:
  • Tách processing logic ra thread pool để consumer main thread kịp poll mới
  • Tăng max.poll.interval.ms (nhưng không quá 30 phút)
  • Giảm max.poll.records để xử lý ít message hơn trong mỗi batch
  • Sử dụng async/await hoặc reactive stream (Project Reactor, RxJava)
  • Thêm timeout cho external calls, không cho phép một call block vô tận
# Cấu hình max.poll.records=50 # Giảm từ default 500 max.poll.interval.ms=300000 # 5 phút (đủ cho processing) # Code pattern: Thread pool + async processing ExecutorService executor = Executors.newFixedThreadPool(5); for (ConsumerRecord record : records) { executor.submit(() -> processMessage(record)); }

2️⃣ Vấn đề về Performance & Throughput

⚡ High Producer Latency
Vấn đề: Độ trễ gửi tin nhắn cao do cấu hình acks=all
⏰ Khi nào xảy ra:
  • Sử dụng acks=all (cần tất cả in-sync replicas acknowledge)
  • Có quá ít in-sync replicas hoặc leader bị chậm
  • Network bị chập chờn giữa các brokers
  • Replication factor quá cao (VD: 5 replicas)
📚 Giải thích về ACK (Acknowledgment):

ACK xác định mức độ "an toàn" của dữ liệu trước khi producer xem message là đã gửi thành công:

  • acks=0 - Producer không đợi phản hồi từ broker. Message được coi là gửi xong ngay lập tức.
    • ⚡ Latency: THẤP NHẤT
    • 💾 Data Loss Risk: CAO NHẤT (nếu broker crash, message bị mất)
    • 📊 Use case: Log, metrics, non-critical data
  • acks=1 - Producer đợi cho đến khi Leader broker acknowledge. Replicas được ghi vào sau đó.
    • ⚡ Latency: TRUNG BÌNH (mất ~10-50ms)
    • 💾 Data Loss Risk: TRUNG BÌNH (nếu leader crash trước khi replicate, message có thể mất)
    • 📊 Use case: Hầu hết applications (recommended cho 90% use cases)
  • acks=all (acks=-1) - Producer đợi cho đến khi tất cả in-sync replicas (ISR) acknowledge. Đây là mode an toàn nhất.
    • ⚡ Latency: CAO NHẤT (phụ thuộc số replicas, có thể 50-200ms+)
    • 💾 Data Loss Risk: THẤP NHẤT (data được ghi trên ≥2 brokers trước khi return success)
    • 📊 Use case: Financial transactions, critical data, compliance required
    • ⚠️ Note: Cần đặt min.insync.replicas ≥ 2 để hiệu quả

💡 Tradeoff: Latency thấp ↔ Data durability cao (chọn một)

✅ Cách giải quyết:
  • Bước 1: Đánh giá yêu cầu data durability của use case
    • Có phải data quan trọng không? (transactions, payments, compliance)
    • Hay chỉ analytics/logs có thể mất một vài message?
  • Nếu cần độ tin cậy cao (Financial, Payment):
    • Sử dụng acks=all với min.insync.replicas=2
    • Replication factor = 3 (1 leader + 2 replicas)
    • Accept latency cao hơn (là đánh đổi cần thiết)
  • Nếu không cần độ tin cậy cao (Logs, Metrics, Analytics):
    • Dùng acks=1 (recommended - balanced choice)
    • Hoặc acks=0 nếu throughput là priority tuyệt đối
    • Latency sẽ giảm 50-80% so với acks=all
  • Tối ưu hóa thêm (nếu acks=all):
    • Tăng batch.size và linger.ms để gom messages thành batch lớn
    • Sử dụng compression (snappy hoặc lz4) để giảm kích thước message
    • Đảm bảo brokers có CPU/IO đủ (scale up hoặc thêm broker)
  • Monitor broker CPU/IO, nâng cấp hardware nếu cần
# Producer config tối ưu cho latency thấp acks=1 # Chỉ cần leader acknowledge batch.size=32768 # 32KB linger.ms=10 # Chờ 10ms để gom batch compression.type=snappy # Nén dữ liệu buffer.memory=67108864 # 64MB buffer
📦 Batch Size Inefficiency
Vấn đề: Kích thước batch quá nhỏ gây tốn tài nguyên mạng, quá lớn gây tốn bộ nhớ
⏰ Khi nào xảy ra:
  • Batch size = 0 (gửi từng message một) → bandwidth lãng phí
  • Batch size quá lớn (VD: 1MB) → producer buffer vỡ, memory spike
  • Network không ổn định → batch bị fragment
  • Không điều chỉnh linger.ms phù hợp
✅ Cách giải quyết:
  • Đặt batch.size = 16KB - 32KB (phù hợp với MTU mạng 1500 bytes)
  • Đặt linger.ms = 10ms - 100ms tùy yêu cầu latency
  • Monitor memory usage của producer, điều chỉnh buffer.memory
  • Kích thước message không nên vượt quá 80% của batch.size
# Balanced config cho throughput & latency batch.size=16384 # 16KB (default is 16KB) linger.ms=50 # Chờ max 50ms hoặc batch full buffer.memory=134217728 # 128MB
🔧 Compression Overhead
Vấn đề: CPU của producer tăng cao khi bật nén (Gzip, Snappy, Zstd)
⏰ Khi nào xảy ra:
  • Sử dụng compression.type=gzip → CPU overhead cao nhất
  • Message rất lớn (> 1MB)
  • Producer chạy trên container với CPU limit thấp
  • Network không phải bottleneck (bandwidth đủ)
✅ Cách giải quyết:
  • Chọn compression phù hợp: snappy (cân bằng), lz4 (nhanh), zstd (nén tốt nhất)
  • Nếu CPU là bottleneck → dùng lz4 hoặc không nén
  • Nếu bandwidth là bottleneck → dùng zstd
  • Tăng CPU resource cho producer container (nếu K8s)
  • Chỉ nén message > 1KB (small messages không nén lợi)
# Recommendation compression.type=snappy # Balanced choice # hoặc compression.type=lz4 # Nếu CPU limited # Performance: snappy > lz4 > gzip (CPU) # Compression: gzip > zstd > snappy > lz4 (Size)

3️⃣ Vấn đề về Kiến trúc & Thiết kế Topic

⚖️ Skewed Partitions
Vấn đề: Một partition chứa quá nhiều dữ liệu so với các partition khác do chọn Key sai
⏰ Khi nào xảy ra:
  • Sử dụng key không đủ diverse (VD: key là region, chỉ có 3 giá trị, 10 partitions)
  • Dùng user_id làm key nhưng một user rất active
  • Dùng null key → tất cả messages ngẫu nhiên vào partition (có thể bị skew)
  • Business logic không cân bằng (VD: 90% users từ 1 region)
✅ Cách giải quyết:
  • Chọn key có entropy cao: user_id, order_id, device_id (không region, status)
  • Sử dụng composite key: region-user_id thay vì chỉ region
  • Nếu cần ordering per key → dùng same key nhưng tăng partition count
  • Monitor partition size qua JMX metrics
  • Nếu đã xảy ra skew → tạo topic mới với partition strategy tốt hơn
# Bad key design String key = record.getRegion(); // Chỉ 3 giá trị → skew # Good key design String key = record.getUserId(); // Millions of unique values # Better - composite key String key = record.getRegion() + "-" + record.getUserId();
🏗️ Inappropriate Retention
Vấn đề: Xóa dữ liệu quá sớm hoặc giữ quá lâu gây tràn đĩa
⏰ Khi nào xảy ra:
  • Retention = 24h → consumer bị lag 1 ngày thì mất dữ liệu
  • Retention = không giới hạn trên topic 10GB/day → 3 tháng = 900GB
  • Broker disk full → Kafka stop accepting writes
  • High throughput topic mà retention không match
✅ Cách giải quyết:
  • Tính toán: throughput (GB/day) × expected max lag (days) = min retention
  • Mặc định: 7 days (1 tuần) là an toàn cho hầu hết use cases
  • High throughput topic (> 1GB/day): 3-7 days
  • Low throughput topic (< 100MB/day): 30+ days
  • Audit logs, compliance data: 1+ năm (lưu trữ riêng)
  • Monitor disk usage: (messages.bytes.total × partition_count) / (1000GB)
# Topic config retention.ms=604800000 # 7 days retention.bytes=-1 # No limit (rely on retention.ms) # Hoặc retention.bytes=107374182400 # 100GB limit retention.ms=-1 # No time limit # Compact topic (keep latest per key) cleanup.policy=compact min.cleanable.dirty.ratio=0.5
🔑 Key Design Issues
Vấn đề: Quên không đặt Key khiến không đảm bảo được thứ tự tin nhắn
⏰ Khi nào xảy ra:
  • producer.send(new ProducerRecord<>(topic, value)) - Key = null
  • Messages được gửi vào random partitions
  • Multiple consumers → thứ tự không guaranteed
  • Cần processing theo thứ tự (VD: user state changes, transactions)
✅ Cách giải quyết:
  • Luôn set key nếu cần ordering: user_id, order_id, account_id
  • Key = null chỉ khi thực sự không cần ordering
  • Sử dụng idempotent producer (enable.idempotence=true) để tránh duplicate khi retry
  • Nếu cần strong ordering: số partitions = 1 (trade-off: throughput)
  • Nếu cần ordering per entity: key = entity_id, multiple partitions OK
# BAD - Key = null producer.send(new ProducerRecord<>(topic, value)); # GOOD - Has key producer.send(new ProducerRecord<>(topic, userId, // Key for ordering messageValue));

4️⃣ Vấn đề về Operational & Infrastructure

💾 Disk Full
Vấn đề: Broker ngừng hoạt động vì hết dung lượng đĩa
⏰ Khi nào xảy ra:
  • Throughput tăng đột ngột nhưng retention không update
  • Retention quá dài (VD: 1 năm) mà không dự tính
  • Log compaction không chạy (compact policy nhưng cleaner thread crash)
  • Không monitor disk space
  • Broker quá cũ, disk quá nhỏ
✅ Cách giải quyết:
  • Ngay lập tức: Tăng retention hoặc xóa topic cũ (backup trước)
  • Thêm alert: disk > 80% → cảnh báo, disk > 90% → page oncall
  • Tính toán disk cần: (topic_throughput_GB_per_day × retention_days) × replication_factor × 1.2 (buffer)
  • Setup cron job để clean up old topics tự động
  • Nâng cấp disk hardware (AWS EBS, add volume)
  • Sử dụng tiered storage (hot/cold) nếu available
# Monitoring script #!/bin/bash USAGE=$(df /var/kafka | awk 'NR==2 {print $5}' | sed 's/%//') if [ $USAGE -gt 90 ]; then echo "CRITICAL: Kafka disk $USAGE% full" | mail admin@company.com fi
📉 In-Sync Replicas (ISR) Shrink
Vấn đề: Danh sách các replica đồng bộ bị thu hẹp, gây rủi ro mất dữ liệu
⏰ Khi nào xảy ra:
  • Broker bị chậm (GC pause, I/O bottleneck) → không kịp replicate
  • Network giữa brokers bị chập chờn
  • replica.lag.time.max.ms quá nhỏ (mặc định 10s)
  • Leader quá tải, chỉ ghi còn không replicate
✅ Cách giải quyết:
  • Tăng replica.lag.time.max.ms (từ 10s lên 30s)
  • Check broker CPU/IO → scale up hardware hoặc số broker
  • Check network bandwidth → upgrade network hoặc move brokers closer
  • Monitor ISR: khi ISR < min.insync.replicas → alert
  • Restart replica broker có vấn đề (graceful restart)
  • Tuning leader: num.network.threads, num.io.threads
# server.properties replica.lag.time.max.ms=30000 # 30 seconds min.insync.replicas=2 num.network.threads=8 num.io.threads=8
⚠️ JVM Garbage Collection (GC) Pauses
Vấn đề: Broker bị treo tạm thời do Java dọn dẹp bộ nhớ (GC pause có thể vài giây)
⏰ Khi nào xảy ra:
  • Broker memory < 2GB (java process phình to)
  • Tệp log quá lớn (> 1GB) → scan lâu
  • Sử dụng G1GC hoặc CMS GC (tuning kém)
  • JVM heap size quá lớn (> 16GB)
✅ Cách giải quyết:
  • Tuning JVM flags: -Xmx8g -Xms8g (8GB for most workloads)
  • Sử dụng ZGC hoặc Shenandoah GC (low-latency)
  • Thêm GC logging để debug: -Xlog:gc*:file=gc.log
  • Monitor GC pauses: nếu > 1s → tune hoặc scale up
  • Avoid: heap > 16GB (GC pause tăng exponential)
# kafka-server-start.sh export KAFKA_HEAP_OPTS="-Xmx8g -Xms8g" export KAFKA_JVM_PERFORMANCE_OPTS=" -XX:+UseZGC -XX:ZCollectionInterval=300000 -XX:+UnlockDiagnosticVMOptions -XX:ZUncommitDelay=300 "

5️⃣ Vấn đề về Data Integrity & Security

🔐 Unencrypted Traffic
Vấn đề: Dữ liệu nhạy cảm truyền qua mạng mà không có SSL
⏰ Khi nào xảy ra:
  • Chạy Kafka trên port 9092 (plaintext) mà không SSL
  • Data contains PII (Personal Identifiable Info): email, phone, SSN
  • Network environment không secure (Internet, shared data center)
  • Compliance requirement (GDPR, HIPAA) không thể ignore
✅ Cách giải quyết:
  • Enable SSL/TLS: cấu hình port 9093 với certificates
  • Generate certificates: openssl hoặc keytool
  • Client config: security.protocol=SSL
  • Encrypt messages tại application level nếu cần double encryption
  • Rotate certificates định kỳ (annual minimum)
  • Monitor certificate expiration: alert 30 days trước hết hạn
# server.properties listeners=PLAINTEXT://0.0.0.0:9092,SSL://0.0.0.0:9093 advertised.listeners=PLAINTEXT://kafka:9092,SSL://kafka:9093 ssl.keystore.location=/etc/kafka/secrets/kafka.server.keystore.jks ssl.keystore.password=your-password ssl.key.password=your-password # client.properties security.protocol=SSL ssl.truststore.location=/etc/kafka/secrets/kafka.client.truststore.jks ssl.truststore.password=your-password
🚫 Authorization Failure
Vấn đề: Cấu hình ACL sai khiến ứng dụng không đọc/ghi được dữ liệu
⏰ Khi nào xảy ra:
  • Enable authorization (authorizer.class.name) nhưng quên cấu hình ACL
  • Cấp quyền sai principal (app-prod vs app-dev)
  • Topic ACL có principal nhưng consumer group khác
  • SASL user không được thêm vào ACL
✅ Cách giải quyết:
  • Kiểm tra ACL: kafka-acls --list
  • Cấp quyền: kafka-acls --create --topic=my-topic --principal=User:app-name --operation=Read,Write
  • Cho consumer group: --operation=Describe --consumer-group=my-group
  • Test authorization với client tương ứng
  • Document who has what access → prevent misconfig
# Grant read + write to app kafka-acls.sh --bootstrap-server localhost:9092 \ --add \ --allow-principal User:app-prod \ --operation Read,Write,Describe \ --topic my-topic \ --consumer-group my-group
📄 Certificate Expiration
Vấn đề: Chứng chỉ SSL hết hạn khiến toàn bộ kết nối bị ngắt
⏰ Khi nào xảy ra:
  • Certificate issuer date 2 năm trước (mặc định hết hạn sau 2 năm)
  • Quên renew trước khi hết hạn
  • Clock skew (server time sai) → certificate bị xem là invalid
  • Production incident: tất cả client disconnect at once
✅ Cách giải quyết:
  • URGENT: Renew certificate immediately (keytool hoặc CA)
  • Setup alert: 30 days trước hết hạn
  • Restart broker sau khi update certificate
  • Check clock sync trên toàn bộ servers: ntpdate, chronyc
  • Use Let's Encrypt with auto-renewal (90 days, auto-renew at 60 days)
  • Implement rolling update để avoid downtime
# Check certificate expiry keytool -list -v -keystore kafka.server.keystore.jks | grep "Valid from" # Auto-renewal setup (cron) 0 0 * * * /opt/certbot/bin/certbot renew --quiet

6️⃣ Vấn đề về Client & App Logic

💧 Resource Leak
Vấn đề: Không đóng Kafka Producer/Consumer khi tắt ứng dụng
⏰ Khi nào xảy ra:
  • Không gọi producer.close() khi app shutdown
  • Consumer quay lòng nhưng connection vẫn open → zombie connection
  • File descriptors cạn kiệt (ulimit)
  • Memory leak → JVM heap size tăng liên tục
✅ Cách giải quyết:
  • Sử dụng try-with-resources (Java): producer/consumer implement AutoCloseable
  • Thêm shutdown hook: Runtime.getRuntime().addShutdownHook()
  • Set timeout: producer.close(30, TimeUnit.SECONDS)
  • Monitor open file descriptors: lsof -p [PID]
  • Check ulimit: ulimit -n (set to 65536 or higher)
// GOOD - try-with-resources try (Producer producer = new KafkaProducer<>(props)) { producer.send(...); } // GOOD - shutdown hook Runtime.getRuntime().addShutdownHook(new Thread(() -> { producer.close(30, TimeUnit.SECONDS); }));
🔁 Infinite Retry
Vấn đề: Producer retry gửi tin nhắn lỗi mãi mãi gây nghẽn hàng đợi
⏰ Khi nào xảy ra:
  • retries = -1 (infinite) mà broker vẫn reject message
  • Message > max.message.bytes → broker luôn reject, producer retry vô tận
  • Network partition kéo dài → producer buffer tính tác
  • Broker down mà producer không có timeout
✅ Cách giải quyết:
  • Set retries = 3 (not -1/infinite)
  • Set request.timeout.ms (mặc định 30s)
  • Implement callback với error handler, log và alert
  • Check message size < max.message.bytes trước gửi
  • Exponential backoff: retry.backoff.ms
  • Send to Dead Letter Topic nếu max retries exceeded
// GOOD - retry config props.put("retries", 3); props.put("request.timeout.ms", 30000); props.put("retry.backoff.ms", 100); // GOOD - error handling producer.send(record, (metadata, exception) -> { if (exception != null) { logger.error("Failed to send: {}", exception); // Send to DLQ or alert } });
🧵 Thread Safety
Vấn đề: Dùng một Consumer instance cho nhiều thread (Kafka Consumer không thread-safe)
⏰ Khi nào xảy ra:
  • Shared consumer across threads → race condition
  • Thread 1 poll, Thread 2 commit → offset quay lùi
  • Crash, data loss, hoặc duplicate processing
  • Rare, non-deterministic → khó debug
✅ Cách giải quyết:
  • ONE consumer per thread pool (rule of thumb)
  • Hoặc: shared consumer nhưng synchronized access (kém performance)
  • Sử dụng reactive stream (Project Reactor, Akka Streams)
  • Spring Kafka handler: single-threaded by default (thread-safe)
  • Code review: kiểm tra consumer usage pattern
// BAD - Shared consumer across threads static KafkaConsumer sharedConsumer = new KafkaConsumer<>(props); // Thread 1 & 2 both poll() → UNSAFE // GOOD - One consumer per thread ExecutorService executor = Executors.newFixedThreadPool(5); for (int i = 0; i < 5; i++) { executor.submit(() -> { KafkaConsumer consumer = new KafkaConsumer<>(props); // ... }); }

7️⃣ Vấn đề về Môi trường Cloud/Kubernetes

🔄 Pod Restart (HPA Continuous Scaling)
Vấn đề: HPA scale liên tục gây rebalance không ngừng
⏰ Khi nào xảy ra:
  • HPA based on CPU/Memory mà threshold quá nhạy cảm
  • CPU 50% → scale up (add pod) → CPU tuột xuống → scale down → repeat
  • Mỗi pod add/remove → Kafka rebalance → tất cả consumer stop
  • Rebalance 30s, processing chậm → CPU cao → scale up lại
✅ Cách giải quyết:
  • Solution 1 (Recommended): Sử dụng KEDA với consumer lag trigger
  • KEDA config: scale up khi lag > 10,000 messages (thay vì CPU 50%)
  • Đảm bảo: maxReplicas = số partitions (không scale vượt quá)
  • Solution 2: HPA stabilization window
  • scaleDownBehavior: stabilization 5 phút trước scale down
  • Solution 3: CooperativeStickyAssignor để rebalance nhanh hơn

📡 Advertising IP/Hostname
Vấn đề: Client bên ngoài không kết nối được vì broker báo địa chỉ IP nội bộ
⏰ Khi nào xảy ra:
  • Kafka in container (K8s): advertised.listeners = pod IP (10.0.0.x)
  • Client outside cluster → có thể reach port 9092 qua service, nhưng metadata trả về pod IP
  • Client try connect to pod IP → fail (no route)
  • Classic K8s issue: không config advertised.listeners đúng
✅ Cách giải quyết:
  • For internal clients: advertised.listeners = service hostname:9092
  • For external clients: advertised.listeners = LoadBalancer IP:port
  • Sử dụng init container để set advertised listeners động
  • Hoặc: helm chart pre-configured cho internal/external
# K8s StatefulSet - init container initContainers: - name: set-advertised-listeners image: kafka:latest command: - sh - -c - | POD_IP=$(hostname -i) sed -i "s/advertised.listeners=.*/advertised.listeners=PLAINTEXT:\/\/$POD_IP:9092/" /etc/kafka/server.properties # Or pass via env (recommended) - name: kafka env: - name: KAFKA_ADVERTISED_LISTENERS value: "PLAINTEXT://kafka-$(POD_ORDINAL).kafka-headless:9092"
⚠️ CPU Throttling
Vấn đề: Pod bị giới hạn CPU khiến xử lý Kafka chậm bất thường
⏰ Khi nào xảy ra:
  • Pod limit CPU = 1 (request = 0.5) → pod throttle từ 50%+ CPU
  • Broker được share node với pods khác → CPU contention
  • Không set CPU requests/limits → pod có thể bị evict khi node full
  • Kafka broker thường cần 2-4 cores
✅ Cách giải quyết:
  • Set CPU requests = actual need (VD: 2 cores)
  • Set CPU limits > requests để pod không bị OOMKill (VD: 3 cores)
  • Sử dụng burstable QoS class (request < limit)
  • Monitor CPU throttling: check cgroup stats
  • Dedicate nodes cho Kafka (node affinity/taints)
  • Thêm buffer: limits = requests + 20%

8️⃣ Các vấn đề khác (Advanced)

⏰ Clock Skew
Vấn đề: Thời gian giữa các máy chủ không đồng bộ gây lỗi timestamp
⏰ Khi nào xảy ra:
  • NTP (Network Time Protocol) không chạy hoặc bị lỗi
  • Message có timestamp tuân theo machine time → bị lỗi logic
  • Timestamp checks (TTL, expiration) fail do clock khác nhau
  • SSL certificate validation fail do server time sai
✅ Cách giải quyết:
  • Enable NTP daemon: systemctl enable ntpd
  • Or use chrony (modern): systemctl enable chronyd
  • Verify sync: timedatectl status (offset < 100ms)
  • Setup monitoring: alert if clock drift > 500ms
  • Use UTC timezone trên tất cả servers
# Check time sync timedatectl status # or ntpstat # Fix if needed chronyc makestep # Force immediate sync
🔗 Connection Storm
Vấn đề: Hàng nghìn client cùng lúc kết nối lại khi Broker vừa khởi động xong
⏰ Khi nào xảy ra:
  • Broker khởi động lại (upgrade, crash recovery)
  • Lúc Broker sẵn sàng, tất cả clients reconnect cùng lúc
  • Broker bị overwhelm → timeout → clients retry → spiral death
  • Especially trong K8s khi scaling up pods
✅ Cách giải quyết:
  • Client-side: thêm random jitter vào reconnect delay
  • Set reconnect.backoff.ms và reconnect.backoff.max.ms
  • Broker: tăng num.network.threads + num.io.threads
  • Rolling restart brokers thay vì restart tất cả cùng lúc
  • Implement backpressure / rate limiting trên broker metadata requests
# Producer/Consumer config reconnect.backoff.ms=50 reconnect.backoff.max.ms=1000 # Broker config - increase threads num.network.threads=8 num.io.threads=8 num.replica.fetchers=4

9️⃣ Vấn đề về Cấu hình Log & Storage (Nâng cao)

🧹 Log Cleaner Thread chết
Vấn đề: Luồng dọn dẹp log bị crash khiến các topic có tính năng compact phình to vô hạn
⏰ Khi nào xảy ra:
  • Topic có cleanup.policy=compact (ví dụ: changelog topics)
  • Log cleaner thread crash → không compact được
  • Disk grows indefinitely → eventually disk full
  • Thường do out of memory hoặc partition corruption
✅ Cách giải quyết:
  • Check broker logs: grep "log.cleaner" logs/server.log
  • Monitor: kafka.log:type=Log,name=CleanerRecycleLatencyMs (JMX)
  • Restart broker nếu cleaner crashed
  • Increase broker memory nếu cần more RAM cho cleaner
  • Tune: log.cleaner.threads, log.cleaner.buffer.size
  • Manual compact nếu cần urgent: kafka-log-dirs tool
# server.properties log.cleaner.threads=1 log.cleaner.buffer.size=536870912 # 512MB log.cleaner.dedupe.buffer.size=134217728 # 128MB log.cleaner.backoff.ms=15000 # Monitor via JMX jconsole kafka:9999 # Check CleanerRecycleLatencyMs
📁 Segment File Locking
Vấn đề: Trên Windows hoặc một số hệ thống file, Kafka không thể xóa các file log cũ do bị process khác lock
⏰ Khi nào xảy ra:
  • Windows filesystem (NTFS) có exclusive file lock
  • Background process (antivirus, backup) giữ file lock
  • Kafka không thể delete old segments → disk grows
  • Rare trên Linux (ext4, xfs), common trên Windows
✅ Cách giải quyết:
  • If on Windows: Disable antivirus for Kafka directories
  • Check for locked files: lsof | grep kafka.log (Linux)
  • Or: Handle.exe -a kafka.log (Windows)
  • Move Kafka to Linux production (avoid Windows)
  • Use SSD + proper filesystem tuning
# Linux - check lock holders lsof +D /var/kafka-logs/ | grep .log # Windows - list handle Handle -a kafka.log

🔟 Vấn đề về Kafka Connect & Ecosystem

📤 Dead Letter Queue (DLQ) phình to
Vấn đề: Quên không monitor DLQ của Kafka Connect khiến nó chiếm hết bộ nhớ hoặc đĩa
⏰ Khi nào xảy ra:
  • Kafka Connect Connector bị lỗi liên tục → send to DLQ
  • DLQ topic không có retention policy
  • Unmonitor → DLQ grows to GB hoặc TB
  • Eventually disk full hoặc memory overload
✅ Cách giải quyết:
  • Set DLQ retention: retention.ms=604800000 (7 days)
  • Setup alert: DLQ lag > 1000 messages
  • Create monitoring dashboard for DLQ size
  • Review DLQ periodically → identify root cause
  • Fix connector or adjust max.retries before DLQ
# Create DLQ topic with retention kafka-topics --create --topic my-connector-dlq \ --partitions 3 \ --replication-factor 3 \ --config retention.ms=604800000 # Monitor DLQ kafka-consumer-groups --group dlq-monitor \ --bootstrap-server kafka:9092 \ --describe
🔀 Source Connector Lag
Vấn đề: Connector đọc từ Database (JDBC) không kịp tiến độ thay đổi của DB
⏰ Khi nào xảy ra:
  • Database có throughput cao nhưng connector chậm
  • JDBC query block → connector chỉ có thể poll chậm chạp
  • Query kỳ vọng 100 rows/sec nhưng DB generate 1000 rows/sec
  • Eventually lag kiếp không kịp theo
✅ Cách giải quyết:
  • Optimize DB query: index, partition, limit rows returned
  • Increase tasks.max (parallel connectors)
  • Use CDC (Change Data Capture) mode instead of polling
  • Upgrade DB → better performance for queries
  • Add monitoring: track connector lag vs DB change rate

1️⃣1️⃣ Vấn đề về Network & Security (Chuyên sâu)

🌐 Wildcard DNS Issues
Vấn đề: Client kết nối nhầm Broker do cấu hình DNS wildcard không chính xác
⏰ Khi nào xảy ra:
  • DNS: *.kafka.internal → resolves to single IP
  • Client connect to kafka-0.kafka.internal → gets wrong Broker
  • Or: all resolve to load balancer IP → no failover
  • Result: some brokers never contacted, hotspot on one broker
✅ Cách giải quyết:
  • Use separate DNS A record per broker (not wildcard)
  • kafka-0.kafka.internal → 10.0.0.1
  • kafka-1.kafka.internal → 10.0.0.2
  • Verify: nslookup kafka-0.kafka.internal
  • Use SRV records for service discovery (if supported)
  • Test client DNS resolution from actual client pod
# Good - individual DNS records kafka-0.kafka.internal A 10.0.0.1 kafka-1.kafka.internal A 10.0.0.2 kafka-2.kafka.internal A 10.0.0.3 # Verify nslookup kafka-0.kafka.internal # Should return 10.0.0.1, NOT load balancer IP
📡 MTU Mismatch
Vấn đề: Kích thước gói tin mạng (MTU) không đồng bộ giữa Client và Broker gây mất gói tin hoặc chậm trễ
⏰ Khi nào xảy ra:
  • Client network: MTU = 9000 (jumbo frames)
  • Broker network: MTU = 1500 (standard)
  • Packets get fragmented or dropped → TCP retransmit
  • Result: latency spike, throughput drop 30-40%
✅ Cách giải quyết:
  • Check MTU: ip link show or ifconfig
  • Ensure all: Client, Broker, Network = same MTU (1500 is standard)
  • Or enable jumbo frames everywhere (9000)
  • Tune batch.size to fit MTU: batch.size < MTU - 50
  • Monitor: check /proc/net/dev for dropped packets
# Check MTU ip link show # or ifconfig eth0 # Set MTU (if needed) ip link set dev eth0 mtu 1500 # Tune Kafka batch for 1500 MTU batch.size=1400 # Leave room for headers

1️⃣2️⃣ Vấn đề về Quản trị & Con người (Operations)

📊 Lack of Monitoring/Alerting
Vấn đề: Hệ thống chạy nhưng không có cảnh báo khi Consumer Lag tăng vọt
⏰ Khi nào xảy ra:
  • Kafka cluster chạy mà không có monitoring setup
  • Consumer lag tăng từ 100 → 100,000 trong 5 phút, không ai biết
  • Broker disk full → chỉ phát hiện khi service down
  • Silent failures → data loss không được phát hiện kịp thời
✅ Cách giải quyết:
  • Setup Prometheus + JMX exporter để collect Kafka metrics
  • Key metrics: consumer lag, broker CPU/IO/disk, replication lag
  • Setup Grafana dashboards for visualization
  • Setup AlertManager rules:
    • Consumer lag > 10k messages (warning), > 100k (critical)
    • Disk > 80% (warning), > 90% (critical)
    • Broker down → page oncall immediately
  • Integrate with PagerDuty / Slack for alerts
# Prometheus alert rule example - alert: KafkaConsumerLagHigh expr: kafka_consumergroup_lag > 100000 for: 5m annotations: summary: "Consumer lag is high: {{ $value }}" - alert: KafkaDiskUsageHigh expr: kafka_broker_disk_used_percent > 90 for: 5m annotations: summary: "Kafka disk usage is critical: {{ $value }}%"
⚠️ Version Mismatch (Client/Server)
Vấn đề: Dùng Client thư viện quá cũ không hỗ trợ các tính năng mới của Broker
⏰ Khi nào xảy ra:
  • Broker = 3.0 (latest), Client = 0.10 (5 năm tuổi)
  • Broker support transactions, client không → compatibility issue
  • Or: broker dropped support for old API version → client error
  • Rare lỗi không dễ debug
✅ Cách giải quyết:
  • Upgrade client library to match broker version
  • Rule: client version ≥ broker version - 2 (for compatibility)
  • Check: kafka-console-producer --version
  • Test before rolling out to production
  • Document supported version matrix in runbook

1️⃣3️⃣ Các lỗi "Góc khuất" (Edge Cases)

⚠️ Leader Not Available
Vấn đề: Toàn bộ Replica của một partition đều bị văng khỏi ISR, không ai có thể làm Leader
⏰ Khi nào xảy ra:
  • Partition replicas = 3, tất cả 3 brokers crash cùng lúc
  • ISR shrink → eventually ISR = empty → no leader
  • Network partition: replicas bị isolated → all out-of-sync
  • Result: partition unavailable, producer/consumer blocked
✅ Cách giải quyết:
  • Check broker status: which brokers are down?
  • Restart crashed brokers ASAP
  • If taking too long, use unclean.leader.election.enable=true (risky - may lose data)
  • Better: set min.insync.replicas=2 + replication.factor=3 to prevent
  • Prevent by: proper monitoring + alerting on ISR shrink
# Emergency - enable unclean leader election (loses data) kafka-configs --entity-type brokers --entity-name 0 \ --alter --add-config unclean.leader.election.enable=true # Better - broker config to prevent unclean.leader.election.enable=false # Default min.insync.replicas=2 # Monitor ISR kafka-topics --describe --topic my-topic \ --bootstrap-server kafka:9092 # Check Isr column - should have >= 2 replicas
💾 Producer Buffer Exhaustion
Vấn đề: Bộ nhớ đệm của Producer bị đầy khiến ứng dụng bị treo (block) khi gọi hàm send()
⏰ Khi nào xảy ra:
  • Producer gửi messages rất nhanh nhưng broker chậm (full load)
  • buffer.memory = 32MB (default) → fill up trong vài giây
  • producer.send() call bị block cho đến khi buffer có space
  • Application thread stuck → làm chậm toàn hệ thống
✅ Cách giải quyết:
  • Increase buffer.memory (từ 32MB → 128MB hoặc 256MB)
  • Implement async sending (not blocking)
  • Add backpressure: pause input if buffer > 80%
  • Scale broker: add more brokers → faster throughput
  • Monitor: track buffer utilization, alert if > 80%
# Increase buffer buffer.memory=268435456 # 256MB (from default 32MB) # Or: async pattern - don't block producer.send(record, (metadata, ex) -> { if (ex != null) handleError(); }); // Don't wait for send to complete
🔄 Graceful Shutdown Failure
Vấn đề: Broker bị tắt cưỡng bức (kill -9) khiến việc khôi phục log khi khởi động lại mất hàng giờ đồng hồ
⏰ Khi nào xảy ra:
  • Broker bị hang → admin chạy kill -9 thay vì graceful shutdown
  • Broker restart → recovery process cần rebuild indices
  • 1TB log → có thể mất 30 phút - 2 giờ để recovery
  • During recovery: broker không available → producer/consumer failed
✅ Cách giải quyết:
  • Always use graceful shutdown: kafka-server-stop.sh (not kill -9)
  • Graceful shutdown: controller notified → leadership transferred → client redirected
  • Set timeout: controlled.shutdown.max.retries=3
  • If needed kill -9: accept recovery time as cost
  • Use rolling restart (one broker at a time) for maintenance
# Graceful shutdown kafka-server-stop.sh # Waits for leader tasks to finish, transfers leadership # Forced stop (last resort) kill -9 [PID] # Will trigger recovery on restart # Rolling restart script for broker in 0 1 2; do kafka-server-stop.sh sleep 30 # Wait for leadership transfer kafka-server-start.sh # Start sleep 120 # Wait to catch up done
