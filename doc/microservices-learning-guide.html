1.1

Microservices là gì?

What is Microservices?
▼

Microservices (vi dịch vụ) là một phong cách kiến trúc phần mềm, trong đó một ứng dụng lớn được chia thành nhiều dịch vụ nhỏ, độc lập, mỗi dịch vụ chỉ làm một việc cụ thể và chạy trong tiến trình riêng của nó.

🍜 Hình dung dễ hiểu

Monolith (Nguyên khối) = Một người đầu bếp làm tất cả: nhận order → nấu ăn → pha đồ uống → làm bánh → tính tiền. Nếu người đó bệnh → cả nhà hàng đóng cửa.

Microservices = Food Court với nhiều gian hàng độc lập: gian phở, gian cơm, gian nước, gian thu ngân — mỗi gian hoạt động độc lập. Gian phở đông khách → thuê thêm người ở gian phở thôi, không cần thuê thêm ở gian nước.

MONOLITH (Nguyên khối)
┌─────────────────────────────────────────┐
│  User UI + Order + Payment + Inventory  │
│  + Notification + Report + Auth + ...   │
│  ← tất cả trong 1 ứng dụng duy nhất →  │
└─────────────────────────────────────────┘

MICROSERVICES (Vi dịch vụ)
┌──────────┐  ┌──────────┐  ┌──────────┐
│  User    │  │  Order   │  │ Payment  │
│ Service  │  │ Service  │  │ Service  │
└──────────┘  └──────────┘  └──────────┘
┌──────────┐  ┌──────────┐  ┌──────────┐
│Inventory │  │Notifica- │  │  Auth    │
│ Service  │  │  tion    │  │ Service  │
└──────────┘  └──────────┘  └──────────┘
Điểm quan trọng cần nhớ
  • Mỗi service chỉ làm một việc (Single Responsibility)
  • Mỗi service có thể được deploy riêng mà không ảnh hưởng service khác
  • Mỗi service có thể dùng ngôn ngữ lập trình khác nhau
  • Các service giao tiếp với nhau qua API hoặc message queue

1.3

Thách thức của Microservices

Challenges — không phải lúc nào cũng là silver bullet
▼

⚠️ Microservices giải quyết một số vấn đề nhưng tạo ra vấn đề mới. Hãy hiểu rõ trade-off trước khi áp dụng.

  • Distributed Systems Complexity (Phức tạp hệ thống phân tán): Network có thể lỗi, service có thể chậm, phải xử lý timeout, retry...
  • Network Latency (Độ trễ mạng): Thay vì gọi function trong cùng process (nanoseconds), giờ phải gọi qua HTTP (milliseconds). Nếu 10 service gọi nhau liên tiếp → latency cộng dồn.
  • Data Consistency (Nhất quán dữ liệu): Không thể dùng database transaction xuyên suốt nhiều service — phải dùng Saga pattern (phức tạp hơn nhiều).
  • Operational Complexity (Phức tạp vận hành): Thay vì deploy 1 app, giờ deploy 20 service + monitor 20 service + logging 20 service...
  • Testing Difficulty (Khó kiểm thử): Làm sao test khi service A phụ thuộc B phụ thuộc C? Cần môi trường phức tạp hơn.

💡 Khi nào NÊN dùng: Team >5 người, hệ thống phức tạp, cần scale một phần độc lập.
Khi nào KHÔNG nên: Startup mới, team nhỏ, domain chưa rõ ràng — bắt đầu với Monolith rồi tách dần.

1.4

Nguyên tắc cốt lõi

Key Principles of Microservices Design
▼

1. Single Responsibility (Một nhiệm vụ duy nhất)

Mỗi service chỉ làm một việc. Nếu bạn mô tả service mà phải dùng chữ "và" → hãy xem xét tách ra.

Ví dụ tốt: "Order Service quản lý đơn hàng" ✓
Ví dụ xấu: "Order Service quản lý đơn hàng và gửi email và tính tiền" ✗

2. Loose Coupling (Liên kết lỏng lẻo)

Các service nên biết ít nhất có thể về nhau. Thay đổi service A không nên bắt service B phải thay đổi theo.

3. High Cohesion (Gắn kết cao)

Những thứ liên quan đến nhau phải nằm trong cùng một service. Code liên quan đến order → Order Service. Không nằm rải rác ở nhiều nơi.

4. Design for Failure (Thiết kế cho sự cố)

Hệ thống phân tán LUÔN có lỗi. Thiết kế từ đầu với giả định rằng service khác có thể không hoạt động. Dùng timeout, retry, circuit breaker, fallback.

5. Decentralized Data (Dữ liệu phi tập trung)

Mỗi service có database riêng. Không chia sẻ database với service khác. Nếu cần data của service khác → gọi API hoặc dùng events.

Domain-Driven Design cơ bản

DDD Basics — Thiết kế theo miền nghiệp vụ
▼

DDD (Domain-Driven Design) = Thiết kế phần mềm xoay quanh nghiệp vụ thực tế, không xoay quanh công nghệ.

🏥 Ví dụ: Hệ thống bệnh viện

Bệnh viện có nhiều "phòng ban" khác nhau: phòng khám, phòng dược, phòng xét nghiệm, phòng tài chính. Mỗi phòng có ngôn ngữ riêng, quy trình riêng — đây chính là Bounded Context.

Mỗi Bounded Context thường tương ứng với một microservice. Khi context nói về "Customer" — trong Order context là người đặt hàng, trong Support context là người cần hỗ trợ — cùng một từ nhưng ý nghĩa khác nhau trong từng context!

Giao tiếp giữa các Service

3.1

Giao tiếp đồng bộ

Synchronous Communication — REST & gRPC
▼
📞 Hình dung

Synchronous = Gọi điện thoại. Bạn gọi → đợi người kia bắt máy → nói chuyện → nhận kết quả ngay. Nếu người kia không bắt → bạn bị block, phải đợi.

REST/HTTP (phổ biến nhất)

const axios = require('axios')

// Order Service gọi Payment Service đồng bộ
async function createOrder(orderData) {
  try {
    // Gọi Payment Service, ĐỢI kết quả trước khi tiếp tục
    const payment = await axios.post('http://payment-service/payments', {
      amount: orderData.total,
      userId: orderData.userId
    }, {
      timeout: 5000  // timeout 5 giây — rất quan trọng!
    })

    return { orderId: 'ORD-001', paymentId: payment.data.id }
  } catch (err) {
    // Phải xử lý lỗi khi service kia không hoạt động
    throw new Error('Payment service unavailable')
  }
}

gRPC — nhanh hơn REST ~5-10x

Dùng Protocol Buffers (định dạng nhị phân) thay vì JSON. Thường dùng cho internal service-to-service communication khi cần hiệu năng cao.

const amqp = require('amqplib') // thư viện RabbitMQ

async function publishOrderCreated(order) {

  const channel = await conn.createChannel()

  const event = {
    type: 'ORDER_CREATED',
    payload: { orderId: order.id, amount: order.total, userId: order.userId },
    timestamp: new Date().toISOString()

  // Gửi message vào queue 'orders' — KHÔNG CẦN ĐỢI ai nhận
  channel.sendToQueue('orders', Buffer.from(JSON.stringify(event)))

async function startConsumer() {
  const conn = await amqp.connect('amqp://rabbitmq')

  // Lắng nghe và xử lý khi có message
  channel.consume('orders', async (msg) => {
    const event = JSON.parse(msg.content.toString())
    if (event.type === 'ORDER_CREATED') {
      await processPayment(event.payload)
      channel.ack(msg) // xác nhận đã xử lý thành công
    }
  })
}

3.4

Circuit Breaker & Fault Tolerance

Cầu dao điện — ngắt mạch khi có sự cố
▼
⚡ Hình dung

Cầu dao điện trong nhà: bình thường điện chạy bình thường (CLOSED). Khi chập điện → cầu dao ngắt (OPEN), bảo vệ thiết bị. Sau một thời gian → thử đóng lại (HALF-OPEN) xem đã sửa chưa. Tương tự: khi service B liên tục lỗi, Circuit Breaker ngắt → service A không gọi B nữa → trả lỗi ngay thay vì đợi timeout.

Trạng thái Circuit Breaker:

CLOSED (Bình thường)  →  OPEN (Ngắt mạch)
   ↑   lỗi < ngưỡng       ↓  lỗi > ngưỡng (vd: 50% fail)
   │                        ↓
HALF-OPEN (Thử lại)  ←  sau 30s timeout
thành công → CLOSED
thất bại  → OPEN lại
// Dùng thư viện opossum (Node.js Circuit Breaker)
const CircuitBreaker = require('opossum')

function callPaymentService(data) {
  return axios.post('http://payment-service/pay', data)
}

const breaker = new CircuitBreaker(callPaymentService, {
  timeout: 3000,        // 3s timeout
  errorThresholdPercentage: 50, // mở khi 50% fail
  resetTimeout: 30000   // sau 30s thử lại (HALF-OPEN)
})

// Fallback: trả kết quả mặc định khi circuit OPEN
breaker.fallback(() => ({ status: 'PENDING', message: 'Sẽ xử lý sau' }))

// Dùng
const result = await breaker.fire({ amount: 100000 })

⚠️ Nhược điểm lớn nhất: Bạn không thể dùng SQL JOIN giữa các service vì chúng nằm trên các database hoàn toàn tách biệt. Có 2 giải pháp chính.

Vấn đề cụ thể — tại sao không dùng JOIN được?

Hãy tưởng tượng bạn muốn lấy danh sách đơn hàng kèm tên người dùng:

-- ❌ KHÔNG làm được: orders DB không biết gì về users DB
SELECT o.id, o.total, u.name, u.email
FROM   orders o
JOIN   users  u ON o.user_id = u.id
-- → ERROR: relation "users" does not exist

-- Vì Order DB chỉ có bảng "orders", không có bảng "users"
-- User DB chỉ có bảng "users", không có bảng "orders"
-- Hai DB hoàn toàn độc lập, không biết nhau

Giải pháp 1: API Composition (Tổng hợp qua API)

🔀 Hình dung

Thay vì JOIN trong DB, bạn tự tay "join" trong code: gọi Order Service lấy đơn hàng, gọi User Service lấy thông tin user, rồi ghép lại trong ứng dụng.

// API Gateway hoặc BFF tự ghép dữ liệu từ nhiều service
async function getOrdersWithUserInfo(userId) {
  // Bước 1: Lấy danh sách orders từ Order Service
  const orders = await orderService.getOrdersByUser(userId)

  // Bước 2: Lấy thông tin user từ User Service
  const user = await userService.getUser(userId)

  // Bước 3: Ghép dữ liệu trong code (thay vì JOIN trong DB)
  return orders.map(order => ({
    ...order,
    user: { name: user.name, email: user.email }
  }))
}

// ⚠️ Vấn đề khi query phức tạp: N+1 Problem
// Nếu user A có 100 đơn hàng, mỗi đơn hàng từ 1 seller khác nhau:
for (const order of orders) {       // 100 orders
  const seller = await sellerService.get(order.sellerId) // 100 HTTP calls!
}
// → 1 request ban đầu + 100 sub-requests = N+1 problem → chậm

Giải pháp 2: CQRS Read Model (Materialized View)

📋 Hình dung

Thay vì JOIN lúc query, bạn pre-join trước khi có event xảy ra, lưu vào một bảng đọc riêng đã được denormalized (phi chuẩn hóa). Giống như bạn photocopy sẵn tất cả thông tin vào một tờ giấy — khi cần chỉ cần lấy tờ đó ra, không cần chạy đi nhiều chỗ ghép lại.

4.2

Distributed Transactions & Eventual Consistency

Giao dịch phân tán & Nhất quán cuối cùng
▼
🏦 Hình dung

Chuyển tiền ngân hàng: trừ tiền tài khoản A (DB ngân hàng A) và cộng tiền tài khoản B (DB ngân hàng B). Với monolith, dùng ACID transaction là xong. Với microservices, 2 DB khác nhau — không thể dùng transaction thông thường!

Saga Pattern — Giải pháp cho distributed transaction

Saga = chuỗi local transactions, mỗi bước khi thất bại thì chạy compensating transaction (giao dịch bù) để hoàn tác các bước trước đó.

💡 Saga có 2 kiểu triển khai: Choreography (mỗi service tự phản ứng với event) và Orchestration (có một "nhạc trưởng" điều phối tất cả). Ví dụ dưới đây dùng Choreography — phổ biến hơn.

Mỗi service lắng nghe event, xử lý, rồi publish event tiếp theo (hoặc compensating event nếu lỗi):

// ── ORDER SERVICE ──────────────────────────────────────────

// Bước 1: Nhận request từ client, tạo order, publish event
app.post('/orders', async (req, res) => {
  const order = await db.query(
    'INSERT INTO orders(user_id, total, status) VALUES($1,$2,$3) RETURNING *',
    [req.user.userId, req.body.total, 'PENDING']
  )
  // Publish event — Payment Service sẽ lắng nghe
  await channel.publish('saga', 'order.created', {
    orderId: order.rows[0].id,
    userId:  req.user.userId,
    total:   req.body.total,
    items:   req.body.items
  })
  res.status(202).json({ orderId: order.rows[0].id, status: 'PENDING' })
})

// Bước 9: Lắng nghe PAYMENT_REFUNDED → compensate: hủy order
channel.consume('payment.refunded', async (event) => {
  await db.query(
    'UPDATE orders SET status=$1 WHERE id=$2',
    ['CANCELLED', event.orderId]
  )
  console.log(`❌ Order ${event.orderId} CANCELLED — reason: ${event.reason}`)
})

// Lắng nghe STOCK_RESERVED → cập nhật order thành CONFIRMED
channel.consume('stock.reserved', async (event) => {
  await db.query(
    'UPDATE orders SET status=$1 WHERE id=$2',
    ['CONFIRMED', event.orderId]
  )
  console.log(`✅ Order ${event.orderId} CONFIRMED`)
})

// ── PAYMENT SERVICE ─────────────────────────────────────────

// Bước 3-4: Lắng nghe ORDER_CREATED → charge tiền → publish tiếp
channel.consume('order.created', async (event) => {
  try {
    // Trừ tiền (gọi payment gateway thực tế)
    const txId = await paymentGateway.charge(event.userId, event.total)

    // Lưu payment record vào DB của Payment Service
    await db.query(
      'INSERT INTO payments(order_id, tx_id, amount, status) VALUES($1,$2,$3,$4)',
      [event.orderId, txId, event.total, 'CHARGED']
    )

    // Thành công → publish để Inventory Service tiếp tục
    await channel.publish('saga', 'payment.done', {
      orderId: event.orderId,
      txId,
      items: event.items
    })
    console.log(`💳 Charged $${event.total} for order ${event.orderId}`)

  } catch (err) {
    // Charge thất bại → compensate: hủy order ngay
    await channel.publish('saga', 'payment.failed', {
      orderId: event.orderId,
      reason:  err.message
    })
  }
})

// Bước 7-8: Lắng nghe STOCK_FAILED → COMPENSATE: hoàn tiền
channel.consume('stock.failed', async (event) => {
  // Lấy txId để hoàn tiền
  const payment = await db.query(
    'SELECT tx_id FROM payments WHERE order_id=$1', [event.orderId]
  )
  // Gọi payment gateway hoàn tiền
  await paymentGateway.refund(payment.rows[0].tx_id)

  // Cập nhật trạng thái payment
  await db.query(
    'UPDATE payments SET status=$1 WHERE order_id=$2',
    ['REFUNDED', event.orderId]
  )

  // Publish để Order Service biết mà hủy order
  await channel.publish('saga', 'payment.refunded', {
    orderId: event.orderId,
    reason:  event.reason
  })
  console.log(`💸 Refunded order ${event.orderId} — reason: ${event.reason}`)
})

// ── INVENTORY SERVICE ───────────────────────────────────────

// Bước 5-6: Lắng nghe PAYMENT_DONE → kiểm tra & trừ tồn kho
channel.consume('payment.done', async (event) => {
  for (const item of event.items) {
    const result = await db.query(
      'UPDATE inventory SET stock = stock - $1 WHERE product_id = $2 AND stock >= $1 RETURNING *',
      [item.qty, item.productId]
    )

    if (result.rowCount === 0) {
      // Hết hàng! → publish STOCK_FAILED để trigger compensate
      await channel.publish('saga', 'stock.failed', {
        orderId: event.orderId,
        reason:  `Product ${item.productId} out of stock`
      })
      console.log(`📦 Stock failed for order ${event.orderId}`)
      return  // dừng lại, không xử lý item tiếp theo
    }
  }

  // Tất cả items đều có đủ hàng → publish SUCCESS
  await channel.publish('saga', 'stock.reserved', {
    orderId: event.orderId
  })
  console.log(`✅ Stock reserved for order ${event.orderId}`)
})

Tại sao cần Idempotency trong Saga?

Nếu message bị deliver 2 lần (network retry), compensating transaction có thể chạy 2 lần → hoàn tiền 2 lần! Giải pháp: mỗi event có idempotency key, consumer kiểm tra trước khi xử lý.

// Trước khi xử lý event, kiểm tra đã xử lý chưa
channel.consume('payment.done', async (event) => {
  const alreadyProcessed = await redis.get(`processed:${event.eventId}`)
  if (alreadyProcessed) {
    channel.ack(msg)  // bỏ qua, đã xử lý rồi
    return
  }

  // Xử lý lần đầu
  await processPayment(event)

  // Đánh dấu đã xử lý (TTL 24h)
  await redis.setEx(`processed:${event.eventId}`, 86400, '1')
  channel.ack(msg)
})

Eventual Consistency (Nhất quán cuối cùng)

Thay vì đảm bảo tất cả DB nhất quán ngay lập tức (strong consistency), hệ thống đảm bảo chúng sẽ nhất quán sau một khoảng thời gian.

📱 Ví dụ thực tế

Facebook Like: bạn like một post → số like hiện lên cho bạn ngay (1,001 likes). Bạn bè ở khắp nơi thế giới vẫn thấy 1,000 likes trong vài giây. Sau đó DB sync → tất cả thấy 1,001. Đây là Eventual Consistency — không phải bug, là thiết kế có chủ đích.

CQRS

Command Query Responsibility Segregation — Tách biệt đọc và ghi
▼
📚 Hình dung

Thư viện có 2 quầy riêng: quầy MƯỢN SÁCH (write — Command) và quầy TRA CỨU/ĐỌC (read — Query). Quầy tra cứu có thể có nhiều bản sao catalog được tối ưu cho tìm kiếm nhanh, trong khi quầy mượn quan tâm đến tính chính xác của dữ liệu.

                    ┌─ Command Side (Ghi) ─┐
Client → API ──┤   Write DB (normalized)  │
                    └──────────────────────┘
                              ↓ publish events
                    ┌─ Query Side (Đọc) ───┐
Client → API ──┤   Read DB (denormalized) │
                    └──────────────────────┘

Command (lệnh ghi): CreateOrder, UpdateOrder, CancelOrder

Query (truy vấn đọc): GetOrderById, ListUserOrders, SearchOrders

💡 Khi nào dùng CQRS: Khi hệ thống có tỉ lệ đọc/ghi chênh lệch lớn (vd: 100 lần đọc / 1 lần ghi), hoặc cần tối ưu query phức tạp mà không ảnh hưởng write performance.

-- Schema của Event Store (PostgreSQL)
CREATE TABLE events (
  id          BIGSERIAL    PRIMARY KEY,
  stream_id   VARCHAR(255) NOT NULL,   -- vd: 'order-ORD-001'
  type        VARCHAR(100) NOT NULL,   -- vd: 'ORDER_CREATED'

-- id | stream_id      | type               | data                           | version | occurred_at

② Order Service — nhận Command, tạo & lưu Event, publish lên Bus

// order-service/src/index.js
app.post('/orders', async (req, res) => {
  const { userId, items } = req.body
  const orderId = `ORD-${Date.now()}`
  const total   = items.reduce((s, i) => s + i.price * i.qty, 0)

  // ② Tạo Domain Event
  const event = {
    streamId:   `order-${orderId}`,
    type:       'ORDER_CREATED',
    data:       { orderId, userId, total, items },
    version:    1,
    occurredAt: new Date().toISOString()
  }

  // ③ Append vào Event Store (ghi vào DB)
  await db.query(
    'INSERT INTO events(stream_id, type, data, version) VALUES($1,$2,$3,$4)',
    [event.streamId, event.type, event.data, event.version]
  )

  // ④ Publish lên Event Bus để các service khác lắng nghe
  await kafka.publish('order-events', event)

  // ⑤ Trả về ngay (202 Accepted) — không đợi payment hay shipping
  res.status(202).json({ orderId, status: 'PENDING', message: 'Order is being processed' })
})

// Rebuild trạng thái order từ events (thay vì SELECT từ orders table)
async function getOrderState(orderId) {
  const { rows } = await db.query(
    'SELECT * FROM events WHERE stream_id=$1 ORDER BY version ASC',
    [`order-${orderId}`]
  )
  // Replay tất cả events → ra trạng thái hiện tại
  return rows.reduce((state, evt) => {
    switch (evt.type) {
      case 'ORDER_CREATED':     return { ...state, ...evt.data, status: 'PENDING' }
      case 'PAYMENT_COMPLETED': return { ...state, status: 'PAID',    txId: evt.data.txId }
      case 'ORDER_SHIPPED':    return { ...state, status: 'SHIPPED', trackingNo: evt.data.trackingNo }
      case 'ORDER_CANCELLED':  return { ...state, status: 'CANCELLED' }
      default: return state
    }
  }, {})
}

    // Publish để Shipping Service lắng nghe tiếp

    console.log(`✅ PAYMENT_COMPLETED published for order ${orderId}`)

  } catch (err) {
    const failEvent = { streamId: `order-${orderId}`, type: 'PAYMENT_FAILED', data: { orderId, reason: err.message }, version: 2 }
    await kafka.publish('order-events', failEvent)
  }
})

Snapshot — tối ưu khi event store quá lớn

Nếu một order có 1000 events, replay 1000 events mỗi lần query sẽ rất chậm. Giải pháp: định kỳ tạo Snapshot (ảnh chụp trạng thái tại một thời điểm), chỉ replay events sau snapshot đó.

// Thay vì replay từ event 1 → 1000
async function getOrderState(orderId) {
  // 1. Lấy snapshot gần nhất
  const snapshot = await db.query(
    'SELECT * FROM snapshots WHERE stream_id=$1 ORDER BY version DESC LIMIT 1',
    [`order-${orderId}`]
  )

  const fromVersion = snapshot.rows[0]?.version ?? 0  // nếu chưa có snapshot → từ đầu
  const initialState = snapshot.rows[0]?.state ?? {}

  // 2. Chỉ replay events SAU snapshot (có thể chỉ 5-10 events)
  const { rows } = await db.query(
    'SELECT * FROM events WHERE stream_id=$1 AND version>$2 ORDER BY version ASC',
    [`order-${orderId}`, fromVersion]
  )

  return rows.reduce((state, evt) => applyEvent(state, evt), initialState)
}

4.5

Cross-Service Query thực tế

Filter + Pagination dữ liệu từ nhiều service — ví dụ Student & Teacher
▼
🎓 Bài toán

Trang Admin có tính năng: Hiển thị danh sách học sinh theo học giáo viên nào — có thể filter theo tên học sinh và tên giáo viên, có pagination. Dữ liệu học sinh nằm ở Student Service, dữ liệu giáo viên nằm ở Teacher Service. Làm sao query hiệu quả?

Vì sao cách đơn giản (API Composition) không work với pagination?

❌ CÁCH NAIVE — BROKEN với Pagination

Admin API nhận request: GET /admin/enrollments?student=A&teacher=B&page=2&size=10

Bước 1: Gọi Student Service → lấy tất cả student tên có "A"
         → [S1, S2, S3, ..., S500]  (500 kết quả)

Bước 2: Với MỖI student, gọi Teacher Service để lấy teacher của họ
         → 500 HTTP calls! (N+1 problem)

Bước 3: Filter chỉ giữ những cặp có teacher tên "B"
         → [S12, S45, S78, S120, S234, S301, ...] (giả sử 47 cặp)

Bước 4: Bây giờ mới paginate: skip 10, take 10
         → [S120, S234, ...]

VẤN ĐỀ:
  ✗ 500 HTTP calls mỗi request → cực kỳ chậm
  ✗ Không biết total count trước khi fetch hết → pagination sai
  ✗ Không scale được (tăng data → tăng calls)
  ✗ Nếu Teacher Service down → cả page down theo

Giải pháp đúng: CQRS Read Model — Materialized View

Xây dựng một bảng riêng (denormalized view) tổng hợp dữ liệu từ cả 2 service. Bảng này được cập nhật tự động khi có event từ Student hoặc Teacher service.

② Query với filter + pagination — cực kỳ đơn giản

// admin-query-service/src/routes/enrollment.js

app.get('/admin/enrollments', async (req, res) => {
  const {
    student = '',   // filter theo tên student
    teacher = '',   // filter theo tên teacher
    page    = 1,
    size    = 10
  } = req.query

  const offset = (page - 1) * size

  // 1 query duy nhất — có filter, có pagination, có total count
  const { rows } = await db.query(`
    SELECT
      student_id, student_name, student_email, student_grade,
      teacher_id, teacher_name, teacher_subject,
      enrolled_at,
      COUNT(*) OVER() AS total_count    -- đếm tổng không cần query thêm
    FROM enrollment_view
    WHERE
      LOWER(student_name) LIKE LOWER($1)   -- filter student tên có "A"
      AND LOWER(teacher_name) LIKE LOWER($2) -- filter teacher tên có "B"
    ORDER BY enrolled_at DESC
    LIMIT $3 OFFSET $4
  `, [`%${student}%`, `%${teacher}%`, size, offset])

  const total = rows[0]?.total_count ?? 0

  res.json({
    data:       rows,
    pagination: {
      page:       Number(page),
      size:       Number(size),
      total:      Number(total),
      totalPages: Math.ceil(total / size),
      hasNext:    page * size < total
    }
  })
  /* Response:
  {
    data: [
      { student_name: "An", teacher_name: "Bình", enrolled_at: "..." },
      { student_name: "Anh", teacher_name: "Bích", enrolled_at: "..." },
      ...
    ],
    pagination: { page: 2, size: 10, total: 47, totalPages: 5, hasNext: true }
  } */

// admin-query-service/src/consumers/student-consumer.js

kafka.consume('student-events', async (event) => {
  switch (event.type) {

    case 'STUDENT_ENROLLED': {
      // Student đăng ký học Teacher → INSERT vào view
      // Nhưng chưa có teacher info → cần fetch từ Teacher Service
      const teacher = await teacherServiceClient.getTeacher(event.data.teacherId)

      await db.query(`
        INSERT INTO enrollment_view
          (student_id, student_name, student_email, student_grade,
           teacher_id, teacher_name, teacher_subject, teacher_email, enrolled_at)
        VALUES ($1,$2,$3,$4, $5,$6,$7,$8, $9)
        ON CONFLICT (student_id, teacher_id) DO NOTHING
      `, [
        event.data.studentId, event.data.studentName, event.data.studentEmail, event.data.grade,
        teacher.id, teacher.name, teacher.subject, teacher.email,
        event.data.enrolledAt
      ])
      break
    }

    case 'STUDENT_UNENROLLED': {
      // Student bỏ học Teacher → DELETE row
      await db.query(
        'DELETE FROM enrollment_view WHERE student_id=$1 AND teacher_id=$2',
        [event.data.studentId, event.data.teacherId]
      )
      break
    }

    case 'STUDENT_UPDATED': {
      // Student đổi tên/email → UPDATE tất cả rows của student đó trong view
      await db.query(`
        UPDATE enrollment_view
        SET student_name=$1, student_email=$2, student_grade=$3, updated_at=NOW()
        WHERE student_id=$4
      `, [event.data.name, event.data.email, event.data.grade, event.data.studentId])
      break
    }
  }
})

async function rebuildEnrollmentView() {
  console.log('🔄 Rebuilding enrollment_view...')

  // 1. Xóa hết data cũ
  await db.query('TRUNCATE enrollment_view')

  // 2. Lấy tất cả enrollments từ Student Service
  let page = 1
  while (true) {
    const { data, hasNext } = await studentService.getAllEnrollments({ page, size: 100 })

      await db.query(`

        teacher.id, teacher.name, teacher.subject, teacher.email,
        enrollment.enrolledAt
      ])
    }

    if (!hasNext) break
    page++
  }

}

💡 Quy tắc chọn approach

Dùng API Composition khi: chỉ cần merge 2-3 objects đơn giản, không có filter cross-service, không cần pagination nghiêm ngặt.

Dùng CQRS Read Model khi: có filter kết hợp nhiều service, cần pagination chính xác, dữ liệu được query thường xuyên, hoặc cần performance cao.

// Truyền qua HTTP header đến tất cả service

// Middleware tạo/truyền Trace ID

  const traceId = req.headers['x-trace-id'] || generateId()
  req.traceId = traceId
  res.set('x-trace-id', traceId)

  // Khi gọi service khác, LUÔN truyền trace ID

Blue-Green Deployment

Hai môi trường — switch traffic không downtime
▼
🎨 Hình dung

Có 2 môi trường giống hệt nhau: Blue (đang chạy v1) và Green (chuẩn bị v2). Deploy v2 vào Green, test kỹ lưỡng. Khi ok → switch toàn bộ traffic sang Green. Rollback = switch lại Blue. Không có thời gian downtime.

Before deploy:
Load Balancer → [Blue: v1] (100% traffic)
               [Green: v1] (idle)

Deploy v2:
Load Balancer → [Blue: v1] (100% traffic)
               [Green: v2] (test trước)

Switch:
Load Balancer → [Blue: v1] (0% - standby để rollback)
               [Green: v2] (100% traffic)

Canary Releases

Triển khai từng bước cho tập user nhỏ trước
▼
🐦 Hình dung

Thợ mỏ ngày xưa mang chim hoàng anh (canary) vào mỏ trước — nếu chim chết, biết có khí độc, người sẽ không vào. Canary release: deploy version mới cho 5% user trước. Nếu không có lỗi → dần tăng lên 25% → 50% → 100%.

Bước 1: v2 deploy cho 5% user  (95% v1, 5% v2)
        → Monitor error rate, latency
Bước 2: Nếu ok → 25% user     (75% v1, 25% v2)
Bước 3: 50% user               (50% v1, 50% v2)
Bước 4: 100% user              (0% v1, 100% v2)
Nếu có vấn đề: rollback về 0% ngay lập tức

  if (!user) return res.status(401).json({ error: 'Sai thông tin' })

// ── Middleware xác thực trong mọi service ──
function authenticate(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1]  // "Bearer <token>"
  if (!token) return res.status(401).json({ error: 'Thiếu token' })
  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET)
    next()
  } catch {
    res.status(401).json({ error: 'Token không hợp lệ' })
  }
}

// ── RBAC: kiểm tra quyền ──
function requireRole(role) {
  return (req, res, next) => {
    if (req.user.role !== role) return res.status(403).json({ error: 'Không có quyền' })
    next()
  }
}

@Injectable()
export class BookingService {
  constructor(
    @InjectRepository(Booking)
    private readonly repo: Repository<Booking>,
    private readonly saga: BookingSaga,
  ) {}

  async initiate(dto: CreateBookingDto) {
    const booking = await this.repo.save(this.repo.create({
      ...dto,
      status:    'PENDING',

    this.saga.execute(booking)

  }

// booking.saga.ts
import { Injectable, Logger } from '@nestjs/common'

@Injectable()
export class BookingSaga {
  private readonly logger = new Logger(BookingSaga.name)

  constructor(
    @InjectRepository(Booking) private readonly repo: Repository<Booking>,
    private readonly inventoryClient: InventoryClient,
    private readonly paymentClient:   PaymentClient,
  ) {}

  async execute(booking: Booking): Promise<void> {
    try {
      await this.lockSeat(booking)       // step 1
      await this.chargePayment(booking)  // step 2
      await this.confirmSeat(booking)    // step 3
      await this.repo.update(booking.id, { status: 'CONFIRMED' })
      this.logger.log(`✅ Booking ${booking.id} CONFIRMED`)
    } catch (err) {
      this.logger.error(`❌ Saga failed: ${err.message}`)
      await this.compensate(booking, err)
      throw err
    }
  }

  private async lockSeat(b: Booking) {
    const locked = await this.inventoryClient.lockSeat(
      { flightId: b.flightId, seatNo: b.seatNo, bookingId: b.id, ttlSeconds: 600 }
    )
    if (!locked) throw new Error('SEAT_UNAVAILABLE')
    await this.repo.update(b.id, {
      status:    'SEAT_LOCKED',
      sagaState: { ...b.sagaState, seatLocked: true },
    })
    b.sagaState = { ...b.sagaState, seatLocked: true }
  }

  private async chargePayment(b: Booking) {
    const result = await this.paymentClient.charge({
      bookingId: b.id, idempotencyKey: b.id,   // idempotency tự động
      amount: b.totalAmount,
    })
    if (!result.success) throw new Error(`PAYMENT_FAILED: ${result.reason}`)
    await this.repo.update(b.id, {
      status:    'PAYMENT_DONE',
      sagaState: { ...b.sagaState, paymentCharged: true, txId: result.txId },
    })
    b.sagaState = { ...b.sagaState, paymentCharged: true, txId: result.txId }
  }

  private async confirmSeat(b: Booking) {
    await this.inventoryClient.confirmSeat(
      { flightId: b.flightId, seatNo: b.seatNo, bookingId: b.id }
    )
  }

  async compensate(b: Booking, err: Error) {
    const { sagaState: s } = b
    if (s?.paymentCharged) await this.paymentClient.refund({ bookingId: b.id, txId: s.txId })
    if (s?.seatLocked)    await this.inventoryClient.releaseSeat(
      { flightId: b.flightId, seatNo: b.seatNo, bookingId: b.id }
    )
    await this.repo.update(b.id, { status: 'CANCELLED' })
  }
}

⚠️ Vấn đề

Chuyến bay còn đúng 1 ghế. 200 người cùng lúc bấm "Book". Nếu không xử lý đúng, tất cả 200 người đều nhận "Đặt thành công" → overbook. Đây là vấn đề khó nhất của hệ thống booking.

CHIẾN LƯỢC: Redis làm "cửa lọc" → PostgreSQL làm "source of truth"

Request 1 (user A)    ───► Redis SETNX seat:VN123:12A:lock  → OK  ✅ → tiếp tục
Request 2 (user B)    ───► Redis SETNX seat:VN123:12A:lock  → FAIL ❌ → trả về "Ghế đã được giữ"
Request 3 (user C)    ───► Redis SETNX seat:VN123:12A:lock  → FAIL ❌ → trả về "Ghế đã được giữ"
...
Request 200 (user Z)  ───► Redis SETNX seat:VN123:12A:lock  → FAIL ❌

Sau 10 phút, user A không thanh toán:
  → Redis TTL expires → key tự xóa → ghế available trở lại
  → User B, C... có thể thử lại
// inventory-service/src/inventory/seat-locker.service.ts
import { Injectable, Logger } from '@nestjs/common'
import { InjectRepository } from '@nestjs/typeorm'
import { Repository } from 'typeorm'
import { InjectRedis } from '@nestjs-modules/ioredis'
import Redis from 'ioredis'
import { Seat } from './entities/seat.entity'

@Injectable()
export class SeatLockerService {
  private readonly logger = new Logger(SeatLockerService.name)

  constructor(
    @InjectRedis() private readonly redis: Redis,
    @InjectRepository(Seat) private readonly seatRepo: Repository<Seat>,
  ) {}

  /** LOCK — atomic SETNX, trả về true nếu thành công */
  async lock(flightId: string, seatNo: string, bookingId: string, ttl = 600): Promise<boolean> {
    const key = `seat:${flightId}:${seatNo}:lock`
    const res = await this.redis.set(key, bookingId, 'NX', 'EX', ttl)
    if (res !== 'OK') return false

    // persist lock vào DB để background job và recovery có thể check
    await this.seatRepo.update(
      { flightId, seatNo, status: 'AVAILABLE' },
      { status: 'LOCKED', lockedBy: bookingId,
        lockedUntil: new Date(Date.now() + ttl * 1000) },
    )
    return true
  }

  /** RELEASE — Lua script atomic, chỉ xóa nếu đúng owner */
  async release(flightId: string, seatNo: string, bookingId: string): Promise<boolean> {
    const lua = `
      if redis.call('GET',KEYS[1])==ARGV[1] then
        return redis.call('DEL',KEYS[1])
      else return 0 end`
    const deleted = await this.redis.eval(lua, 1, `seat:${flightId}:${seatNo}:lock`, bookingId) as number
    if (deleted) await this.seatRepo.update({ flightId, seatNo }, { status: 'AVAILABLE', lockedBy: null, lockedUntil: null })
    return deleted === 1
  }

  /** CONFIRM — xóa Redis lock + UPDATE DB status=BOOKED (optimistic) */
  async confirm(flightId: string, seatNo: string, bookingId: string): Promise<void> {
    await this.release(flightId, seatNo, bookingId)
    const { affected } = await this.seatRepo
      .createQueryBuilder()
      .update(Seat)
      .set({ status: 'BOOKED', lockedBy: null, lockedUntil: null })
      .where('flightId=:f AND seatNo=:s AND lockedBy=:b AND status=:st',
             { f: flightId, s: seatNo, b: bookingId, st: 'LOCKED' })
      .execute()
    if (!affected) throw new Error('CONFIRM_FAILED: seat state mismatch')
  }
}

import { Injectable } from '@nestjs/common'
import { ElasticsearchService } from '@nestjs/elasticsearch'

@Injectable()
export class SearchService {
  constructor(private readonly es: ElasticsearchService) {}

  async searchFlights(dto: SearchFlightDto) {
    const priceField = dto.cabin === 'BUSINESS' ? 'business_price' : 'economy_price'
    const { hits } = await this.es.search({
      index: 'flights',
      from:  (dto.page - 1) * dto.size,
      size:  dto.size,
      query: {
        bool: {
          must: [
            { term: { origin: dto.from } },
            { term: { destination: dto.to } },
            { range: { departs_at: { gte: `${dto.date}T00:00:00`, lte: `${dto.date}T23:59:59` } } },
            { term: { status: 'ACTIVE' } },
            { range: { available_seats: { gt: 0 } } },
            ...(dto.maxPrice ? [{ range: { [priceField]: { lte: dto.maxPrice } } }] : []),
          ],
        },
      },
      sort: [{ [priceField]: 'asc' }, { departs_at: 'asc' }],
    })
    return { data: hits.hits.map(h => h._source), total: (hits.total as any).value }
  }

  async updateAvailableSeats(flightId: string, available: number) {
    await this.es.update({ index: 'flights', id: flightId, doc: { available_seats: available } })
  }

  async updatePrice(flightId: string, economy: number, business: number) {
    await this.es.update({ index: 'flights', id: flightId, doc: { economy_price: economy, business_price: business } })
  }
}

PRICING RULES (ví dụ đơn giản):
  > 50% seats available  → base price

// pricing-service/src/pricing/pricing.service.ts
import { Injectable, Logger } from '@nestjs/common'
import { EventPattern, Payload, Client, ClientKafka } from '@nestjs/microservices'
import { InjectRepository } from '@nestjs/typeorm'
import { Repository } from 'typeorm'
import { FlightPrice } from './entities/flight-price.entity'

@Injectable()
export class PricingService {
  private readonly logger = new Logger(PricingService.name)

  constructor(
    @InjectRepository(FlightPrice) private readonly priceRepo: Repository<FlightPrice>,
    @Client({ transport: 'KAFKA', options: { client: { brokers: ['kafka:9092'] } } })
    private readonly kafka: ClientKafka,
  ) {}

  // Lắng nghe event từ Inventory Service mỗi khi seat bị book/release
  @EventPattern('inventory.changed')
  async onInventoryChanged(
    @Payload() d: { flightId: string; totalSeats: number; availableSeats: number }
  ) {
    const ratio      = d.availableSeats / d.totalSeats
    const basePrice  = await this.getBasePrice(d.flightId)
    const multiplier = ratio > 0.5  ? 1.00
                     : ratio > 0.3  ? 1.10
                     : ratio > 0.1  ? 1.25
                     : d.availableSeats > 5 ? 1.50 : 2.00
    const newPrice = Math.round(basePrice * multiplier)

    await this.priceRepo.update({ flightId: d.flightId }, { economyPrice: newPrice })

    // Publish để Search Service cập nhật Elasticsearch index
    this.kafka.emit('pricing.updated', {
      flightId: d.flightId, economy: newPrice
    })
    this.logger.log(`💰 Flight ${d.flightId} → ${newPrice} (x${multiplier})`)
  }

  private async getBasePrice(flightId: string): Promise<number> {
    const price = await this.priceRepo.findOneBy({ flightId })
    return price?.basePrice ?? 500_000
  }
}

const jwt      = require('jsonwebtoken')
const bcrypt   = require('bcryptjs')
const { Pool } = require('pg')

const app = express()

const db = new Pool({ connectionString: process.env.DB_URL })

// Tạo bảng khi khởi động
db.query(`CREATE TABLE IF NOT EXISTS users (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,
  password VARCHAR(255) NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
)`)

// POST /api/auth/register
app.post('/register', async (req, res) => {
  const { email, password } = req.body
  const hashed = await bcrypt.hash(password, 10)
  try {
    const { rows } = await db.query(
      'INSERT INTO users(email, password) VALUES($1,$2) RETURNING id,email',
      [email, hashed]
    )
    res.status(201).json(rows[0])
  } catch (e) {
    res.status(400).json({ error: 'Email đã tồn tại' })
  }
})

// POST /api/auth/login
app.post('/login', async (req, res) => {
  const { email, password } = req.body
  const { rows } = await db.query('SELECT * FROM users WHERE email=$1', [email])
  const user = rows[0]
  if (!user || !await bcrypt.compare(password, user.password))
    return res.status(401).json({ error: 'Sai thông tin đăng nhập' })

  const token = jwt.sign({ userId: user.id, email }, process.env.JWT_SECRET, { expiresIn: '1h' })
  res.json({ token })
})

app.listen(3001, () => console.log('User Service running on :3001'))

4. order-service/src/index.js

const express  = require('express')
const jwt      = require('jsonwebtoken')
const amqp     = require('amqplib')
const { Pool } = require('pg')

const app = express()
app.use(express.json())
const db = new Pool({ connectionString: process.env.DB_URL })
let channel  // RabbitMQ channel

// Khởi tạo kết nối RabbitMQ
async function initRabbitMQ() {
  const conn = await amqp.connect(process.env.RABBITMQ_URL)
  channel = await conn.createChannel()
  await channel.assertExchange('orders', 'topic', { durable: true })
  console.log('✓ Kết nối RabbitMQ thành công')
}

// Middleware xác thực JWT
function auth(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1]
  if (!token) return res.status(401).json({ error: 'Chưa đăng nhập' })
  try { req.user = jwt.verify(token, process.env.JWT_SECRET); next() }
  catch { res.status(401).json({ error: 'Token không hợp lệ' }) }
}

// Tạo bảng orders
db.query(`CREATE TABLE IF NOT EXISTS orders (
  id SERIAL PRIMARY KEY,
  user_id INTEGER NOT NULL,
  items JSONB NOT NULL,
  total NUMERIC NOT NULL,
  status VARCHAR(50) DEFAULT 'PENDING',
  created_at TIMESTAMP DEFAULT NOW()
)`)

// POST /api/orders — Tạo đơn hàng
app.post('/', auth, async (req, res) => {
  const { items } = req.body
  if (!items?.length) return res.status(400).json({ error: 'Giỏ hàng trống' })

  const total = items.reduce((s, i) => s + i.price * i.qty, 0)
  const { rows } = await db.query(
    'INSERT INTO orders(user_id,items,total) VALUES($1,$2,$3) RETURNING *',
    [req.user.userId, JSON.stringify(items), total]
  )
  const order = rows[0]

  // Publish event ORDER_CREATED vào RabbitMQ
  channel.publish('orders', 'order.created',
    Buffer.from(JSON.stringify({
      type: 'ORDER_CREATED',
      orderId: order.id,
      userId: order.user_id,
      total: order.total
    }))
  )
  console.log(`✓ Order ${order.id} created, event published`)
  res.status(201).json(order)
})

// GET /api/orders
app.get('/', auth, async (req, res) => {
  const { rows } = await db.query(
    'SELECT * FROM orders WHERE user_id=$1 ORDER BY created_at DESC',
    [req.user.userId]
  )
  res.json(rows)
})

async function start() {
  await initRabbitMQ()
  app.listen(3002, () => console.log('Order Service running on :3002'))
}
start()

5. payment-service/src/consumer.js

const amqp = require('amqplib')
const express = require('express')
const app = express()

async function startConsumer() {
  const conn    = await amqp.connect(process.env.RABBITMQ_URL)
  const channel = await conn.createChannel()

  await channel.assertExchange('orders', 'topic', { durable: true })
  const { queue } = await channel.assertQueue('payment-queue', { durable: true })
  await channel.bindQueue(queue, 'orders', 'order.created')

  console.log('Payment Service: Đang lắng nghe order.created events...')

  channel.consume(queue, async (msg) => {
    const event = JSON.parse(msg.content.toString())
    console.log(`💳 Xử lý thanh toán: Order #${event.orderId}, Amount: ${event.total}`)

    // Giả lập xử lý thanh toán (thực tế sẽ gọi payment gateway)
    await new Promise(r => setTimeout(r, 1000))

    // Publish PAYMENT_COMPLETED event
    channel.publish('orders', 'payment.completed',
      Buffer.from(JSON.stringify({
        type: 'PAYMENT_COMPLETED',
        orderId: event.orderId,
        userId: event.userId
      }))
    )
    console.log(`✓ Thanh toán thành công: Order #${event.orderId}`)
    channel.ack(msg)  // Xác nhận đã xử lý thành công
  })
}

// Health check endpoint
app.get('/health', (_, res) => res.json({ status: 'ok' }))
app.listen(3003)
