Introduction to Microservices

Understand why microservices exist, the benefits they offer, and the core principles behind the architecture.

1.1

What are Microservices?

The shift from Monolith to independent services
▼

Microservices is an architectural style where a large application is broken down into small, independently deployable services. Each service does one specific thing, runs in its own process, and communicates with others via APIs or messaging.

🍔 Real-world analogy

Monolith = A single chef who takes orders, cooks food, makes drinks, bakes desserts, and handles billing. If the chef gets sick, the whole restaurant shuts down.

Microservices = A food court with separate stalls — burgers, sushi, drinks, checkout. Each stall operates independently. When the burger stall gets busy, you hire more burger staff — not more sushi chefs.

MONOLITH (Single deployable unit)
┌──────────────────────────────────────────────┐
│  Users + Orders + Payments + Inventory       │
│  + Notifications + Reports + Auth + Search   │
│  ← everything in one codebase & process →   │
└──────────────────────────────────────────────┘

MICROSERVICES (Many independent services)
┌──────────┐  ┌──────────┐  ┌──────────┐
│  User    │  │  Order   │  │ Payment  │
│ Service  │  │ Service  │  │ Service  │
└──────────┘  └──────────┘  └──────────┘
┌──────────┐  ┌──────────┐  ┌──────────┐
│Inventory │  │Notif-    │  │  Auth    │
│ Service  │  │ication   │  │ Service  │
└──────────┘  └──────────┘  └──────────┘
Key Takeaways
  • Each service does one thing only (Single Responsibility)
  • Each service can be deployed independently without touching others
  • Each service can use a different programming language or database
  • Services communicate via APIs or message queues

1.3

Benefits & Challenges

Microservices is not a silver bullet — know the trade-offs
▼

⚠️ Microservices solve some problems but create new ones. Understand the trade-offs before adopting.

  • Distributed Systems Complexity: Network can fail. Services can be slow. You must handle timeouts, retries, and partial failures.
  • Network Latency: Calling a function in the same process takes nanoseconds. Calling a service over HTTP takes milliseconds. With 10 chained calls, latency compounds.
  • Data Consistency: You can't use a database transaction across multiple services — you need the Saga pattern (much more complex).
  • Operational Overhead: Instead of deploying 1 app, you deploy, monitor, and log 20+ services.
  • Testing Complexity: Testing when Service A depends on B depends on C requires a more involved environment.

💡 When to use: Large teams, complex domain, need to scale parts independently.
When NOT to use: Early-stage startup, small team, unclear domain — start with a Monolith, extract services gradually.

1.4

Key Principles of Microservices Design

The rules that make microservices work well
▼

1. Single Responsibility

Each service does one thing. If you need the word "and" to describe what a service does — consider splitting it.

Good: "Order Service manages orders" ✓
Bad: "Order Service manages orders and sends emails and processes payments" ✗

2. Loose Coupling

Services should know as little as possible about each other. Changing Service A should not force Service B to change.

3. High Cohesion

Things related to each other should live in the same service. All order-related logic belongs in the Order Service — not scattered across multiple services.

4. Design for Failure

In a distributed system, failures will happen. Design assuming that other services can be unavailable. Use timeouts, retries, circuit breakers, and fallbacks.

5. Decentralized Data Management

Each service owns its database. No shared databases between services. If a service needs another's data — call its API or use events.

Synchronous Communication

REST & gRPC — caller waits for a response
▼
📞 Analogy

Synchronous = A phone call. You dial → wait for the other person to pick up → talk → get an answer immediately. If they don't pick up, you're blocked waiting.

const axios = require('axios')

// Order Service calls Payment Service synchronously
async function createOrder(orderData) {
  try {
    // We WAIT for the payment response before continuing
    const payment = await axios.post('http://payment-service/payments', {
      amount: orderData.total,
      userId: orderData.userId

      timeout: 5000  // always set a timeout!

    return { orderId: 'ORD-001', paymentId: payment.data.id }
  } catch (err) {
    // Must handle the case where the other service is down
    throw new Error('Payment service unavailable')
  }
}

Message Queues & Event Streaming — fire and forget
▼
📬 Analogy

Asynchronous = Sending an email. You write it → hit send → go back to work immediately. You don't wait for the recipient to read and reply. They'll process it when they're ready.

                         (Middleman)

                                         →→→    Notification Service

// ── Publisher: Order Service publishes an event ──
const amqp = require('amqplib')

async function publishOrderCreated(order) {
  const conn    = await amqp.connect('amqp://rabbitmq')
  const channel = await conn.createChannel()

  const event = {
    type:      'ORDER_CREATED',
    payload:   { orderId: order.id, amount: order.total, userId: order.userId },
    timestamp: new Date().toISOString()
  }

  // Send to queue — we do NOT wait for anyone to consume it
  channel.sendToQueue('orders', Buffer.from(JSON.stringify(event)))
  console.log('Event published, moving on...')
}

// ── Consumer: Payment Service listens ──
async function startConsumer() {
  const conn    = await amqp.connect('amqp://rabbitmq')
  const channel = await conn.createChannel()
  await channel.assertQueue('orders')

  channel.consume('orders', async (msg) => {
    const event = JSON.parse(msg.content.toString())
    if (event.type === 'ORDER_CREATED') {
      await processPayment(event.payload)
      channel.ack(msg) // acknowledge successful processing
    }
  })
}

Circuit Breakers & Fault Tolerance

Stop calling a broken service, fail fast
▼
⚡ Analogy

A circuit breaker in your home: normally electricity flows freely (CLOSED). When there's a short circuit → the breaker trips (OPEN), protecting your devices. After some time → you try to reset it (HALF-OPEN). Same idea: when Service B keeps failing, the Circuit Breaker trips — Service A stops calling B and returns an error immediately instead of waiting for timeout.

Circuit Breaker States:

CLOSED (Normal) →  OPEN (Tripped)
   ↑ errors below threshold    ↓ errors exceed threshold (e.g. 50%)
   │                           ↓
HALF-OPEN (Testing) ← after 30s timeout
  success → CLOSED
  failure → OPEN again
// Using 'opossum' — Node.js Circuit Breaker library
const CircuitBreaker = require('opossum')

function callPaymentService(data) {
  return axios.post('http://payment-service/pay', data)
}

const breaker = new CircuitBreaker(callPaymentService, {
  timeout:                  3000,  // 3s timeout per call
  errorThresholdPercentage: 50,    // open when 50% fail
  resetTimeout:             30000  // try again after 30s
})

// Fallback: return a safe default when circuit is OPEN
breaker.fallback(() => ({ status: 'PENDING', message: 'Will be processed shortly' }))

const result = await breaker.fire({ amount: 100 })

Data Management

Each service owns its data — so how do we keep data consistent across services?

4.1

Database per Service Pattern

Each service owns its own database
▼

The rule: each microservice owns its data and is the only one with direct database access. Other services must go through its API to get data.

❌ Wrong: Shared Database
User Service  ─┐
Order Service ─┤→ [Shared Database]
Payment Svc   ─┘
→ Schema change in one place → all services potentially break

✓ Right: Database per Service
User Service   → [User DB — PostgreSQL]
Order Service  → [Order DB — PostgreSQL]
Payment Svc    → [Payment DB — MySQL]
Search Svc     → [Search DB — Elasticsearch]
Session Svc    → [Session DB — Redis]

Each service can also choose the database type best suited for its use case!

⚠️ Biggest downside: You cannot use SQL JOINs across services because they live in completely separate databases. There are two primary solutions.

The concrete problem — why JOINs don't work

Say you want to list orders along with the customer's name:

-- ❌ IMPOSSIBLE: Order DB has no knowledge of the Users DB
SELECT o.id, o.total, u.name, u.email
FROM   orders o
JOIN   users  u ON o.user_id = u.id
-- → ERROR: relation "users" does not exist

-- Because:
-- Order DB  only has the "orders" table
-- User DB   only has the "users"  table
-- They are completely isolated — neither knows the other exists

Solution 1: API Composition

🔀 Analogy

Instead of letting the database do the join, you do the join in application code: call Order Service for orders, call User Service for user info, then merge the results yourself.

// API Gateway or BFF merges data from multiple services
async function getOrdersWithUser(userId) {
  // Step 1: Fetch orders from Order Service
  const orders = await orderService.getOrdersByUser(userId)

  // Step 2: Fetch user info from User Service
  const user = await userService.getUser(userId)

  // Step 3: Merge in code (the "JOIN" happens in application layer)
  return orders.map(order => ({
    ...order,
    user: { name: user.name, email: user.email }
  }))
}

// ⚠️ Watch out for the N+1 Problem on complex queries:
// If user A has 100 orders, each from a different seller:
for (const order of orders) {          // 100 orders
  const seller = await sellerService.get(order.sellerId) // 100 HTTP calls!
}
// → 1 original request + 100 sub-requests = very slow

Solution 2: CQRS Read Model (Materialized View)

📋 Analogy

Instead of joining at query time, you pre-join when events happen and store the merged result in a dedicated read-optimized table. Like photocopying all the relevant information onto one sheet in advance — when someone asks, you just hand them the sheet. No running around gathering pieces.

// ── Step 1: Each service publishes events when data changes ──

// Order Service: publish when an order is created
channel.publish('events', 'order.created', {
  orderId: 1, userId: 42, total: 199, status: 'PENDING'
})

// User Service: publish when a user updates their profile
channel.publish('events', 'user.updated', {
  userId: 42, name: 'Alice', email: 'alice@example.com'
})

// ── Step 2: Read Model Service listens, builds a denormalized "view" ──
channel.consume('order.created', async (event) => {
  // Fetch extra info once — at event time, not at query time
  const user = await userService.getUser(event.userId)

  // Store into a READ-optimized table — already denormalized
  await readDb.query(`
    INSERT INTO order_summary
      (order_id, user_id, user_name, user_email, total, status)
    VALUES ($1, $2, $3, $4, $5, $6)
  `, [event.orderId, event.userId, user.name, user.email, event.total, 'PENDING'])
})

// ── Step 3: Lightning-fast queries — no JOINs needed ──
app.get('/order-summary', async (req, res) => {
  // Single simple query — all data is already co-located
  const { rows } = await readDb.query(
    'SELECT * FROM order_summary WHERE user_id = $1', [req.user.userId]
  )
  // → Result already contains: order_id, user_name, user_email, total, status
  res.json(rows)
})

4.2

Distributed Transactions & Eventual Consistency

Saga Pattern — the solution to cross-service transactions
▼
🏦 Analogy

Wire transfer: debit Account A (Bank A's DB) and credit Account B (Bank B's DB). In a monolith, one ACID transaction handles this. In microservices, two separate databases — you can't span a single transaction across them!

Saga Pattern — the solution

A Saga is a sequence of local transactions. Each step publishes an event triggering the next. If a step fails, compensating transactions undo all previous committed steps — in reverse order.

💡 Sagas come in two flavors: Choreography (each service reacts to events independently) and Orchestration (a central Saga Orchestrator drives the flow). The examples below use Choreography — the more common approach.

Code — Choreography Saga implementation

Each service listens for events, does its local work, then publishes the next event (or a compensating event on failure):

// ── ORDER SERVICE ──────────────────────────────────────────

// Step 1-2: Receive client request, create order, publish event
app.post('/orders', async (req, res) => {
  const order = await db.query(
    'INSERT INTO orders(user_id, total, status) VALUES($1,$2,$3) RETURNING *',
    [req.user.userId, req.body.total, 'PENDING']
  )
  // Kick off the saga — Payment Service will pick this up
  await channel.publish('saga', 'order.created', {
    orderId: order.rows[0].id,
    userId:  req.user.userId,
    total:   req.body.total,
    items:   req.body.items
  })
  res.status(202).json({ orderId: order.rows[0].id, status: 'PENDING' })
})

// Step 9: Compensate — cancel order when payment was refunded
channel.consume('payment.refunded', async (event) => {
  await db.query(
    'UPDATE orders SET status=$1 WHERE id=$2',
    ['CANCELLED', event.orderId]
  )
  console.log(`❌ Order ${event.orderId} CANCELLED — reason: ${event.reason}`)
})

// Happy path: confirm order when stock is reserved
channel.consume('stock.reserved', async (event) => {
  await db.query(
    'UPDATE orders SET status=$1 WHERE id=$2',
    ['CONFIRMED', event.orderId]
  )
  console.log(`✅ Order ${event.orderId} CONFIRMED`)
})

// ── PAYMENT SERVICE ─────────────────────────────────────────

// Steps 3-4: Listen for ORDER_CREATED → charge → publish next
channel.consume('order.created', async (event) => {
  try {
    const txId = await paymentGateway.charge(event.userId, event.total)

    await db.query(
      'INSERT INTO payments(order_id, tx_id, amount, status) VALUES($1,$2,$3,$4)',
      [event.orderId, txId, event.total, 'CHARGED']
    )
    // Payment succeeded → let Inventory Service continue
    await channel.publish('saga', 'payment.done', {
      orderId: event.orderId, txId, items: event.items
    })
    console.log(`💳 Charged $${event.total} for order ${event.orderId}`)

  } catch (err) {
    // Charge failed → compensate immediately
    await channel.publish('saga', 'payment.failed', {
      orderId: event.orderId, reason: err.message
    })
  }
})

// Steps 7-8: COMPENSATE — refund when stock reservation failed
channel.consume('stock.failed', async (event) => {
  const payment = await db.query(
    'SELECT tx_id FROM payments WHERE order_id=$1', [event.orderId]
  )
  // Issue the refund
  await paymentGateway.refund(payment.rows[0].tx_id)
  await db.query(
    'UPDATE payments SET status=$1 WHERE order_id=$2',
    ['REFUNDED', event.orderId]
  )
  // Notify Order Service to cancel
  await channel.publish('saga', 'payment.refunded', {
    orderId: event.orderId, reason: event.reason
  })
  console.log(`💸 Refunded order ${event.orderId} — reason: ${event.reason}`)
})

// ── INVENTORY SERVICE ───────────────────────────────────────

// Steps 5-6: Listen for PAYMENT_DONE → check and deduct stock
channel.consume('payment.done', async (event) => {
  for (const item of event.items) {
    const result = await db.query(
      'UPDATE inventory SET stock = stock - $1 WHERE product_id=$2 AND stock >= $1 RETURNING *',
      [item.qty, item.productId]
    )
    if (result.rowCount === 0) {
      // Out of stock! → trigger compensating chain
      await channel.publish('saga', 'stock.failed', {
        orderId: event.orderId,
        reason:  `Product ${item.productId} out of stock`
      })
      console.log(`📦 Stock failed for order ${event.orderId}`)
      return  // stop processing further items
    }
  }
  // All items in stock → saga happy path complete
  await channel.publish('saga', 'stock.reserved', { orderId: event.orderId })
  console.log(`✅ Stock reserved for order ${event.orderId}`)
})

Why idempotency is critical in Sagas

If a message is delivered twice (network retry), a compensating transaction could run twice — refunding money twice! Solution: each event carries an idempotency key; consumers check before processing.

// Check idempotency before processing any event
channel.consume('payment.done', async (event) => {
  const alreadyDone = await redis.get(`processed:${event.eventId}`)
  if (alreadyDone) { channel.ack(msg); return }  // skip duplicate

  await processStock(event)

  await redis.setEx(`processed:${event.eventId}`, 86400, '1')  // mark done, 24h TTL
  channel.ack(msg)
})

Eventual Consistency

Instead of guaranteeing all databases are in sync immediately (strong consistency), the system guarantees they will be consistent eventually.

📱 Real-world example

Facebook Like: you like a post → your count shows 1,001 immediately. Your friend in Tokyo still sees 1,000 for a few seconds. After DB sync → everyone sees 1,001. This is Eventual Consistency — a deliberate design choice, not a bug.

4.3

CQRS

Command Query Responsibility Segregation — separate read and write models
▼
📚 Analogy

A library has two separate desks: a CHECKOUT desk (write — Commands) and a CATALOG desk (read — Queries). The catalog can have multiple optimized copies for fast search, while checkout focuses on accuracy. They serve different purposes with different requirements.

                    └──────────────────────────┘

  • Commands (writes): CreateOrder, UpdateOrder, CancelOrder
  • Queries (reads): GetOrderById, ListUserOrders, SearchOrders

② Order Service — receive Command, create & store Event, publish to Bus

// order-service/src/index.js
app.post('/orders', async (req, res) => {
  const { userId, items } = req.body
  const orderId = `ORD-${Date.now()}`
  const total   = items.reduce((s, i) => s + i.price * i.qty, 0)

  // ② Produce the Domain Event
  const event = {
    streamId:   `order-${orderId}`,
    type:       'ORDER_CREATED',
    data:       { orderId, userId, total, items },
    version:    1,
    occurredAt: new Date().toISOString()
  }

  // ③ Append to Event Store (write to DB)
  await db.query(
    'INSERT INTO events(stream_id, type, data, version) VALUES($1,$2,$3,$4)',
    [event.streamId, event.type, event.data, event.version]
  )

  // ④ Publish onto Event Bus for other services to consume
  await kafka.publish('order-events', event)

  // ⑤ Return immediately — don't wait for payment or shipping
  res.status(202).json({ orderId, status: 'PENDING', message: 'Order is being processed' })
})

// Rebuild order state from events (instead of SELECT from orders table)
async function getOrderState(orderId) {
  const { rows } = await db.query(
    'SELECT * FROM events WHERE stream_id=$1 ORDER BY version ASC',
    [`order-${orderId}`]
  )
  // Replay all events → current state
  return rows.reduce((state, evt) => {
    switch (evt.type) {
      case 'ORDER_CREATED':     return { ...state, ...evt.data, status: 'PENDING'   }
      case 'PAYMENT_COMPLETED': return { ...state, status: 'PAID',    txId: evt.data.txId }
      case 'ORDER_SHIPPED':    return { ...state, status: 'SHIPPED', trackingNo: evt.data.trackingNo }
      case 'ORDER_CANCELLED':  return { ...state, status: 'CANCELLED' }
      default: return state
    }
  }, {})
}

  switch (event.type) {
    case 'ORDER_CREATED':
      await readDb.query(`
        INSERT INTO order_view(order_id, user_id, total, status, items, created_at)
        VALUES($1, $2, $3, $4, $5, $6)
      `, [orderId, event.data.userId, event.data.total, 'PENDING', JSON.stringify(event.data.items), event.occurredAt])
      break

    case 'PAYMENT_COMPLETED':
      await readDb.query(`
        UPDATE order_view SET status='PAID', tx_id=$1, paid_at=$2 WHERE order_id=$3
      `, [event.data.txId, event.occurredAt, orderId])
      break

    case 'ORDER_SHIPPED':
      await readDb.query(`
        UPDATE order_view SET status='SHIPPED', tracking_no=$1, shipped_at=$2 WHERE order_id=$3
      `, [event.data.trackingNo, event.occurredAt, orderId])
      break

    case 'ORDER_CANCELLED':
      await readDb.query(`
        UPDATE order_view SET status='CANCELLED', cancelled_at=$1 WHERE order_id=$2
      `, [event.occurredAt, orderId])
      break
  }
  // After every event → order_view always reflects the latest state
})

Snapshots — optimization for large event streams

If an order accumulates 1,000 events, replaying all of them on every read is slow. Solution: periodically save a Snapshot (the state at that point), then only replay events after the snapshot.

// Instead of replaying events 1 → 1000...
async function getOrderState(orderId) {
  // 1. Fetch the most recent snapshot
  const snapshot = await db.query(
    'SELECT * FROM snapshots WHERE stream_id=$1 ORDER BY version DESC LIMIT 1',
    [`order-${orderId}`]
  )
  const fromVersion  = snapshot.rows[0]?.version ?? 0
  const initialState = snapshot.rows[0]?.state    ?? {}

  // 2. Only replay events AFTER the snapshot (maybe just 5-10 events)
  const { rows } = await db.query(
    'SELECT * FROM events WHERE stream_id=$1 AND version>$2 ORDER BY version ASC',
    [`order-${orderId}`, fromVersion]
  )
  return rows.reduce((state, evt) => applyEvent(state, evt), initialState)
}

Cross-Service Queries in Practice

Filter + Pagination across multiple services — Student & Teacher example
▼
🎓 The Problem

The Admin dashboard has a feature: Show a list of students and which teachers they are enrolled with — filterable by student name and teacher name, with pagination. Student data lives in Student Service, teacher data lives in Teacher Service. How do you query this efficiently?

Why the naive approach (API Composition) breaks with pagination

❌ NAIVE APPROACH — BROKEN with Pagination

Admin API receives: GET /admin/enrollments?student=A&teacher=B&page=2&size=10

Step 1: Call Student Service → get all students whose name contains "A"
         → [S1, S2, S3, ..., S500]  (500 results)

Step 2: For EACH student, call Teacher Service to get their teacher
         → 500 HTTP calls! (N+1 problem)

Step 3: Filter to keep only pairs where teacher name contains "B"
         → [S12, S45, S78, S120, S234, S301, ...] (say 47 pairs)

Step 4: Now paginate: skip 10, take 10
         → [S120, S234, ...]

PROBLEMS:
  ✗ 500 HTTP calls per request → extremely slow
  ✗ Can't know total count without fetching everything → pagination is wrong
  ✗ Doesn't scale (more data → more calls)
  ✗ If Teacher Service is down → the whole page is down

Correct solution: CQRS Read Model — Materialized View

Build a dedicated denormalized view table that aggregates data from both services. This table is automatically updated whenever events arrive from either the Student or Teacher service.

ARCHITECTURE OVERVIEW

② Query with filter + pagination — dead simple

// admin-query-service/src/routes/enrollment.js

app.get('/admin/enrollments', async (req, res) => {
  const {
    student = '',   // filter by student name
    teacher = '',   // filter by teacher name
    page    = 1,
    size    = 10
  } = req.query

  const offset = (page - 1) * size

  // Single query — filters, pagination, and total count all in one
  const { rows } = await db.query(`
    SELECT
      student_id, student_name, student_email, student_grade,
      teacher_id, teacher_name, teacher_subject,
      enrolled_at,
      COUNT(*) OVER() AS total_count    -- total without an extra query
    FROM enrollment_view
    WHERE
      LOWER(student_name) LIKE LOWER($1)
      AND LOWER(teacher_name) LIKE LOWER($2)
    ORDER BY enrolled_at DESC
    LIMIT $3 OFFSET $4
  `, [`%${student}%`, `%${teacher}%`, size, offset])

  const total = rows[0]?.total_count ?? 0

  res.json({
    data:       rows,
    pagination: {
      page:       Number(page),
      size:       Number(size),
      total:      Number(total),
      totalPages: Math.ceil(total / size),
      hasNext:    page * size < total
    }
  })
  /* Response:
  {
    data: [
      { student_name: "Alice", teacher_name: "Bob", enrolled_at: "..." },
      ...
    ],
    pagination: { page: 2, size: 10, total: 47, totalPages: 5, hasNext: true }
  } */
})

        teacher.id, teacher.name, teacher.subject, teacher.email,

// JWT = Header.Payload.Signature (base64 encoded)
// eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOiIxMjMifQ.abc123

// ── Auth Service: Issue JWT on login ──
app.post('/login', async (req, res) => {
  const user = await validateCredentials(req.body)
  if (!user) return res.status(401).json({ error: 'Invalid credentials' })

  const token = jwt.sign(
    { userId: user.id, role: user.role },  // payload
    process.env.JWT_SECRET,               // secret key
    { expiresIn: '15m' }               // short-lived access token
  )
  res.json({ accessToken: token })
})

// ── JWT verification middleware (reuse in every service) ──
function authenticate(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1]  // "Bearer <token>"
  if (!token) return res.status(401).json({ error: 'Missing token' })
  try { req.user = jwt.verify(token, process.env.JWT_SECRET); next() }
  catch { res.status(401).json({ error: 'Invalid token' }) }
}

// ── RBAC: restrict by role ──
const requireRole = (role) => (req, res, next) => {
  if (req.user.role !== role) return res.status(403).json({ error: 'Forbidden' })
  next()
}

class BookingSaga {
  async execute(bookingId, input) {
    const booking = await Booking.create({
      ...input, status: 'PENDING',
      sagaState: { step: 'INIT', seatLocked: false, paymentCharged: false }
    })
    try {
      await this.lockSeat(booking)
      await this.chargePayment(booking)
      await this.confirmSeat(booking)
      await booking.update({ status: 'CONFIRMED' })
      await kafka.publish('booking-events', { type: 'BOOKING_CONFIRMED', data: booking })
      return booking
    } catch (err) {
      await this.compensate(booking, err)
      throw err
    }
  }

  async lockSeat(booking) {
    const ok = await inventoryClient.lockSeat({
      flightId: booking.flightId, seatNo: booking.seatNo,
      bookingId: booking.id, ttlSeconds: 600
    })
    if (!ok) throw new Error('SEAT_UNAVAILABLE')
    await booking.update({ status: 'SEAT_LOCKED', sagaState: { ...booking.sagaState, step: 'SEAT_LOCKED', seatLocked: true } })
  }

  async chargePayment(booking) {
    const result = await paymentClient.charge({
      bookingId: booking.id, idempotencyKey: booking.id,
      amount: booking.totalAmount, userId: booking.userId
    })
    if (!result.success) throw new Error('PAYMENT_FAILED: ' + result.reason)
    await booking.update({ status: 'PAYMENT_PENDING', sagaState: { ...booking.sagaState, step: 'PAYMENT_DONE', paymentCharged: true, txId: result.txId } })
  }

  async compensate(booking, err) {
    console.error(`[Saga] Compensating booking ${booking.id}, failed at: ${err.message}`)
    const { sagaState } = booking
    if (sagaState.paymentCharged) await paymentClient.refund({ bookingId: booking.id, txId: sagaState.txId })
    if (sagaState.seatLocked)    await inventoryClient.releaseSeat({ flightId: booking.flightId, seatNo: booking.seatNo, bookingId: booking.id })
    await booking.update({ status: 'CANCELLED', cancelReason: err.message })
  }
}

Inventory Service — Overbooking Prevention

Redis atomic SETNX + Optimistic Locking: 200 users booking seat 12A simultaneously
▼
⚠️ The Problem

One seat left. 200 users press "Book" at the same time. Without proper handling, all 200 get "Booking successful" → overbooking. This is the hardest problem in any booking system.

STRATEGY: Redis as "gate filter" → PostgreSQL as source of truth

// inventory-service/src/seatLocker.js
class SeatLocker {
  async lock(flightId, seatNo, bookingId, ttlSeconds = 600) {
    const key = `seat:${flightId}:${seatNo}:lock`
    // SET key value NX EX ttl — atomic: only sets if key doesn't exist
    const result = await redis.set(key, bookingId, 'NX', 'EX', ttlSeconds)
    return result === 'OK'
  }

  async release(flightId, seatNo, bookingId) {
    const key = `seat:${flightId}:${seatNo}:lock`
    // Lua script: check ownership & delete atomically
    const lua = `
      if redis.call('GET', KEYS[1]) == ARGV[1] then
        return redis.call('DEL', KEYS[1])
      else return 0 end`
    return (await redis.eval(lua, 1, key, bookingId)) === 1
  }

  async confirm(flightId, seatNo, bookingId) {
    await this.release(flightId, seatNo, bookingId)
    // Optimistic lock: only update if this booking still holds the seat
    const { rowCount } = await db.query(`
      UPDATE seats SET status='BOOKED', locked_by=NULL, locked_until=NULL, version=version+1
      WHERE flight_id=$1 AND seat_no=$2 AND locked_by=$3 AND status='LOCKED'
    `, [flightId, seatNo, bookingId])
    if (rowCount === 0) throw new Error('CONFIRM_FAILED: seat state changed')
  }
}

// Background job: release expired seats every minute (safety net)
setInterval(async () => {
  await db.query(`

// pricing-service/src/pricingEngine.js
kafka.consume('inventory-events', async (event) => {
  if (event.type !== 'INVENTORY_CHANGED') return
  const { flightId, totalSeats, availableSeats } = event.data
  const ratio = availableSeats / totalSeats
  const basePrice = await getBasePrice(flightId)
  const multiplier = ratio > 0.5  ? 1.00
                   : ratio > 0.3  ? 1.10
                   : ratio > 0.1  ? 1.25
                   : availableSeats > 5 ? 1.50 : 2.00
  const newPrice = Math.round(basePrice * multiplier)
  await db.query('UPDATE flight_prices SET economy_price=$1 WHERE flight_id=$2', [newPrice, flightId])
  await kafka.publish('pricing-events', { type: 'PRICE_UPDATED', data: { flightId, economyPrice: newPrice } })
})

app.post('/login', async (req, res) => {
  const { email, password } = req.body
  const { rows } = await db.query('SELECT * FROM users WHERE email=$1', [email])
  const user = rows[0]
  if (!user || !await bcrypt.compare(password, user.password))
    return res.status(401).json({ error: 'Invalid credentials' })
  const token = jwt.sign({ userId: user.id, email }, process.env.JWT_SECRET, { expiresIn: '1h' })
  res.json({ token })
})

function authenticate(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1]
  if (!token) return res.status(401).json({ error: 'Unauthorized' })
  try { req.user = jwt.verify(token, process.env.JWT_SECRET); next() }
  catch { res.status(401).json({ error: 'Invalid token' }) }
}

db.query(`CREATE TABLE IF NOT EXISTS orders (
  id SERIAL PRIMARY KEY, user_id INTEGER NOT NULL,
  items JSONB NOT NULL, total NUMERIC NOT NULL,
  status VARCHAR(50) DEFAULT 'PENDING', created_at TIMESTAMP DEFAULT NOW()
)`)

app.post('/', authenticate, async (req, res) => {
  const { items } = req.body
  if (!items?.length) return res.status(400).json({ error: 'Cart is empty' })
  const total = items.reduce((s, i) => s + i.price * i.qty, 0)
  const { rows } = await db.query(
    'INSERT INTO orders(user_id,items,total) VALUES($1,$2,$3) RETURNING *',
    [req.user.userId, JSON.stringify(items), total]
  )
  const order = rows[0]
  channel.publish('orders', 'order.created',
    Buffer.from(JSON.stringify({ type: 'ORDER_CREATED', orderId: order.id, userId: order.user_id, total: order.total }))
  )
  console.log(`✓ Order ${order.id} created and event published`)
  res.status(201).json(order)
})

  const { rows } = await db.query('SELECT * FROM orders WHERE user_id=$1 ORDER BY created_at DESC', [req.user.userId])
